---

### M1: 運用安定化（M0 後 〜 2 週間）

**目標**: 実運用のフィードバックを集めて致命的な問題を潰す。機能追加はしない。

| 観点 | 確認項目 |
|---|---|
| 入力速度 | CaaF 3 タップ以内で完了するか（IC-3 閾値） |
| LLM 精度 | 抽出の信号色分布（🟢率 vs 🟡🟠率） |
| 一覧の実用性 | フィルタ・検索で目的の工具に辿り着けるか |
| 返却フロー | クイック返却の操作ミスがないか |
| PWA 体験 | ホーム追加で起動できるか、表示崩れはないか |
| マスタ網羅性 | 現場の工具がマスタに足りているか |

**余白**: この期間は**機能を足さない**。運用の声だけ聞く。2 週間は最低確保。早く安定すれば M2 に進む。

---

### M2: 改善ループ（M1 後 〜 自然なペースで）

**目標**: M1 のフィードバックを反映し、統合準備を並行で進める。

#### 改善候補（運用で優先度が決まる）
- LLM プロンプトチューニング（抽出精度向上）
- エイリアス学習の運用改善
- マスタ管理 UI の使い勝手向上
- パフォーマンス改善（一覧表示速度等）

#### 統合準備（ハンドオフ文書 §9 対応）
| 項目 | 内容 |
|---|---|
| Server Action 戻り値型 | `ActionResult<T>` パターンに統一 |
| emoji アイコン置換 | bottom-tabs の 📋✏️🔧 → lucide 系 |
| 速報イベント選定 | Activity Feed に流す 3-5 イベントを確定 |
| INTEGRATION-PLAN.md 完成 | 骨子 → 運用実績を踏まえて肉付け |

**余白**: 改善と統合準備の比率は固定しない。運用が安定していれば統合準備を厚く、問題が多ければ改善を厚く。**期限は設けず、「使える」状態を維持しながら進める**。

---

### M3: Platform 統合（Platform v3.0 受入セッション）

**前提条件**（ハンドオフ文書より）:
- 独立リポで本番稼働実績があること
- SPEC / REGIME / INDEX / spec/ が最新・正確であること
- §9 チェックリスト全項目 PASS

| 領域 | 作業 |
|---|---|
| A. コード移植 | `apps/platform/app/apps/tools/` に物理配置 |
| B. 認証切替 | Gate 画面 → `employees.profile_id = auth.uid()` に置換 |
| C. 共有パッケージ | `@kakuman/supabase` / `@kakuman/ui` に切替 |
| D. 接続点 | Activity Feed 投入 + ホーム launcher タイル |

**卒業判定**: 統合完了後、独立デプロイを停止するか 2 系統並走するかは M2 の運用実績で判断。

## 設計原則（ロードマップ全体を通して不変）

- 罠 A 群は絶対遵守（状態カラム禁止、append-only）
- SSOT 優先順位: `.ts > .sql > .md`
- ヒト > モノ > コト の設計階層
- Phase 0 = 全件確認（自動 INSERT は Phase 1 以降）
- 入力速度の劣化は失敗

# F2 Chat UI v2 — ストローク対話モデル

## 概要

1メッセージの入力から登録完了までを **1ストローク** と定義し、
ストローク内で文脈を保持したまま修正・確認・確定を対話で完結させる。

現行の「一括カード → 全件確定 or 全件修正」を廃止し、
**個別アイテム確認 + ストローク内対話修正** に移行する。

## 現行 → 新規の比較

### 現行フロー（v1）

```
User: "バッテリー 2,3番 ブロワ 1番 池下現場"
System: [1枚の信号色カード（全アイテム一括表示）]
        → 問題があれば全体がブロック
User: [確定] or [修正]（修正=テキストに戻って最初からやり直し）
```

**問題点:**
- 5工具中1つだけ番号ミスでも全体がブロックされる
- 修正すると抽出結果が破棄される（やり直し）
- 「3番→1番に変えて」のような部分修正ができない

### 新フロー（v2 ストローク対話モデル）

```
User: "バッテリー 2,3番 ブロワ 1番 池下現場"
System: 池下現場 — 3件を確認してください
        ┌──────────────────────────────┐
        │ ✅ マキタバッテリーV18 6 #2    [確定] │
        │ ❌ マキタバッテリーV18 6 #3    [修正] │  ← 存在しない番号
        │    利用可能: 1,4,5,6,7...番          │
        │ ✅ 充電ブロワー MUB363D #1    [確定] │
        └──────────────────────────────┘
        [全件確定]

User: "3番は1番に変更"               ← ストローク内の対話修正
System: #3 → #1 に変更しました
        ┌──────────────────────────────┐
        │ ✅ マキタバッテリーV18 6 #2    [確定] │
        │ ✅ マキタバッテリーV18 6 #1    [確定] │  ← 修正反映
        │ ✅ 充電ブロワー MUB363D #1    [確定] │
        └──────────────────────────────┘
        [全件確定]

User: [全件確定]
System: ✅ 3件 登録完了
```

## 設計原則

### ストロークとは

- **1ストローク = 1回の入力から登録完了までの対話の塊**
- ストロークは開始（ユーザー送信）と終了（全件確定/スキップ or キャンセル）を持つ
- ストローク内では文脈（現場名、action、抽出結果）が保持される
- ストローク完了後、次のストロークに持ち越さない（リセット）
- 複数ストロークを並行させない（1ストロークずつ処理）

### 制約遵守

- D-4: Phase 0 = 全件人間確認。「全件確定」も人間タップ必須
- D-5: LLM は修正指示の解釈のみ。欠損情報を創作しない
- D-8: スワイプ/ロングプレス禁止。タップのみ
- 3タップ: 入力 → 確認 → 確定（ストロークは「確認」ステップを対話的に拡張する）

## アイテム個別確認

### アイテムの状態遷移

```
pending (解析完了)
  ├→ confirmed (個別確定 or 全件確定)
  ├→ skipped (スキップ)
  ├→ clarifying (修正中 → 再解決 → pending に戻る)
  └→ error (登録失敗)
```

### 表示ルール

| アイテム状態 | バッジ色 | アクション |
|---|---|---|
| matched（全一致） | 緑 | [確定] [スキップ] |
| unit_missing（番号不在） | 赤 | [修正] [スキップ] — 確定不可 |
| not_found（マスタ未一致） | 赤 | [修正] [スキップ] — 確定不可 |
| no_unit_specified（番号未指定） | 黄 | [番号選択] [スキップ] |
| confirmed（確定済み） | グレー + チェック | (操作不可) |
| skipped（スキップ済み） | グレー + 取消線 | [取消] で pending に戻す |

### 全件確定ボタン

- pending かつ matched のアイテムが1件以上あるときのみ有効
- unit_missing / not_found のアイテムは全件確定に含まれない（個別修正が必要）
- 確定対象アイテム数を表示: 「N件を確定」

## ストローク内対話修正

### 修正指示の処理

ストロークがアクティブ（reviewing状態）のとき、ユーザーの追加入力は
**修正指示** として解釈される。LLM に以下のコンテキストを渡す:

```
元の入力: {originalText}
現在の抽出結果: {extraction}
ユーザーの修正指示: {clarificationText}
```

LLM は修正指示を解釈し、抽出結果の該当部分を更新する。

### 修正指示の例

| ユーザー入力 | 解釈 | アクション |
|---|---|---|
| 「3番は1番に変更」 | unit_number 3 → 1 | 該当アイテムの番号を変更して再解決 |
| 「ブロワはスキップ」 | ブロワをスキップ | 該当アイテムを skipped に |
| 「現場は星ヶ丘」 | 現場名変更 | extraction.site を更新 |
| 「バッテリーは4番も追加」 | アイテム追加 | 新アイテムを解決して追加 |
| 「やっぱり全部やめる」 | キャンセル | ストローク終了（何も登録しない） |

### 修正後の再解決

番号変更 → `resolveAgainstMaster` を該当アイテムのみ再実行。
全体の信号色を再計算。

## ストロークのデータモデル

```typescript
type ItemStatus = "pending" | "confirmed" | "skipped" | "clarifying" | "error";

type StrokeItem = {
  /** resolveAgainstMaster の結果 */
  resolved: ResolvedItem;
  /** 個別アイテムの状態 */
  status: ItemStatus;
};

type StrokeMessage = {
  role: "user" | "system";
  content: string;
};

type Stroke = {
  id: string;
  /** ストローク内の全メッセージ（対話履歴） */
  messages: StrokeMessage[];
  /** LLM 抽出結果（修正で更新される） */
  extraction: CaaFExtractionResult | null;
  /** 全体の信号色 */
  signal: Signal | null;
  /** 個別アイテムの確認状態 */
  items: StrokeItem[];
  /** ストローク全体の状態 */
  phase: "extracting" | "reviewing" | "saving" | "done" | "cancelled" | "error";
  /** 登録結果 */
  result: { insertedCount: number; skippedCount: number; errors: string[] } | null;
};
```

## 入力バーの動的変化

| ストローク状態 | placeholder | 送信ボタン |
|---|---|---|
| なし（idle） | 「工具名・番号・現場を入力...」 | 送信アイコン |
| reviewing | 「修正内容を入力...」 | 送信アイコン |
| saving | (入力不可) | — |

## 番号未指定時の番号選択 UI

`no_unit_specified` のアイテムには、利用可能な番号をタップ可能なバッジとして
横並びで表示する。タップで選択 → pending に戻る。

```
充電式インパクトドライバ (番号を選択してください)
 [1] [2] [4] [5] [7]     ← タップ可能バッジ
```

## スクロール挙動

チャットエリアの自動スクロール（末尾追従）は **ユーザーが送信したときだけ** 行う。
ストローク内の状態更新では表示位置を固定し、勝手に動かさない。

| トリガー | 挙動 |
|---|---|
| 新規入力の送信 / 修正指示の送信 | 末尾へスクロール（自分の発言を表示） |
| 候補・番号・案件の選択 | 位置維持 |
| セレクタ（候補/番号ピッカー）の表示切替 | 位置維持 |
| 確定 / スキップ / 全件確定 | 位置維持 |
| 解析完了に伴う再レンダ（extracting→reviewing） | 位置維持 |

**理由**: モバイルで選択・確認のたびに画面が最下部へジャンプすると、確認中の
アイテムを見失い操作性を著しく損なう。送信は明示的ユーザー操作なので末尾追従して
よいが、選択・確認・解析完了などストローク内の更新では位置を固定する。

**実装注意**: 末尾スクロールを `strokes` 変化全般に反応させると本問題が再発する。
スクロールは送信ハンドラが明示要求したときのみ実行する（`useCaaF` の送信フラグ駆動）。

## 完了時の表示

```
✅ 3件 登録完了 / 1件 スキップ
  マキタバッテリーV18 6 #2  ✓
  マキタバッテリーV18 6 #1  ✓
  充電ブロワー MUB363D #1  ✓
  充電式インパクトドライバ  —（スキップ）
```

## UX Must 閾値

| 指標 | 閾値 | 備考 |
|---|---|---|
| 初回抽出（LLM 応答） | p95 5秒以内 | 既存 F2 と同じ |
| 修正指示の反映 | p95 3秒以内 | LLM 再解釈 + 部分再解決 |
| 確定から登録完了 | p95 2秒以内 | DB INSERT |
| 全体のストローク完了 | 問題なければ 3タップ | 入力 → (確認) → 全件確定 |

## DONT（追加）

- ストローク間の文脈引き継ぎはしない（1ストローク=独立）
- 修正指示の LLM 解釈で元の入力にない工具を創作しない（D-5 維持）
- 個別確定を自動化しない（D-4 維持、Phase 0 = 全件人間タップ）
- 「全部確定して」の音声/テキスト指示で自動全件確定しない（「全件確定」ボタン必須）
- 選択・確認・解析完了など送信以外の操作で画面位置を勝手に動かさない（末尾スクロールは送信時のみ）

## 段階的実装ロードマップ

### Step 1: 個別アイテム確認（最小変更）
- SignalCard → ItemCard のリスト表示に変更
- 各アイテムに [確定] [スキップ] ボタン
- [全件確定] ボタン
- 修正はまだテキスト入力→最初からやり直し

### Step 2: ストローク内対話修正
- ストロークモデル導入
- 入力バーの動的 placeholder
- LLM に修正指示を解釈させる clarification prompt
- 部分再解決

### Step 3: 番号選択 UI
- no_unit_specified 時のタップ可能番号バッジ
- unit_missing 時の代替番号サジェスト

## 既存制約との整合

| 制約 | 整合 |
|---|---|
| 罠A | 不変。状態は View 導出のまま |
| D-1 (append-only) | 不変。確定=INSERT、スキップ=何もしない |
| D-3 (confidence) | 不変。信号色は表示のみ |
| D-4 (Phase 0) | 強化。個別確認でさらに人間判断が細かくなる |
| D-5 (LLM抽出器) | 維持。修正解釈も抽出の延長、創作はしない |
| D-7 (holder/moved_by) | 不変 |
| D-8 (グローブ操作) | 不変。全操作がタップ |

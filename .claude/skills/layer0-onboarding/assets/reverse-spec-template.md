# 逆仕様化（Reverse Spec）テンプレート

既存コードから SPEC.md / INTENT.md を逆生成するためのテンプレート。
onboarding スキルの処理フロー 4 で使用する。

---

## 基本方針

- **as-is 中心**: 既存の挙動を忠実に記述。改善提案は **to-be セクション** に分離
- **F 番号採番**: Archaeology で抽出した機能を F1 から順に採番。優先順位・参照順序は後続で調整
- **確度メタデータ必須**: 全機能に **コード確認 / git 履歴確認 / AI 推定** のいずれかを付与
- **凍結線と連動**: FZ-N（凍結線）と重なる機能はその旨を明記

---

## SPEC.md（onboarding 版）構造

```markdown
# 仕様（onboarding 由来）

## 概要

[プロジェクトの目的・対象ユーザー・主要価値を 3〜5 行で記述]

## as-is 機能一覧（既存挙動の記録）

### F1: [機能名]

- **意図**: [なぜ存在するか。git log / README から読み取れた「なぜ」]
- **条件**: [入力・前提条件]
- **振る舞い**: [事後条件・出力]
- **実装パス**: `src/xxx.ts:100-150`
- **確度**: コード確認 / git 履歴確認 / AI 推定 (2026-04-19)
- **凍結線**: 振る舞い凍結（デフォルト） / FZ-1 コード凍結（該当時）
- **優先度**: 高 / 中 / 低（AI 推定 — 呼び出し頻度から）

### F2: ...

## to-be 候補（改善提案、未適用）

[onboarding では実装しない。spec-architect へ引き継いで人間判断]

### T1: [提案タイトル]

- **発見内容**: Archaeology 中に検出した改善余地
- **提案**: [to-be の具体化]
- **根拠**: [なぜ提案するか]
- **影響範囲**: [関連 F 番号]

## 抽出不能領域

[DONT.md にも記載する。SPEC.md では抽出不能だった事実のみ記録]

- F7 タイムアウト値 3000ms の根拠不明
- 外部 API X の応答スキーマ未確定

## 機密領域（存在のみ記録）

- `.env.production` — 内容は抽出対象外
- `config/oauth-secrets.yaml` — 内容は抽出対象外
```

## INTENT.md（history/ 配下）への展開

onboarding で抽出した意図は `history/INTENT.md` に F 番号ごとに記録する。
history-layer-spec.md のフォーマットに準拠。

```markdown
# 機能意図履歴

## F1: [機能名]

- **意図**: [なぜこの機能か]
- **条件**: [主要条件]
- **優先度**: 高 / 中 / 低
- **確度**: **AI 推定** (2026-04-19, onboarding 由来)
  — 後続 spec-architect で人間確認時に「人間確定」へ昇格
- **onboarding 時の原典**: git log sha1:abc123, README.md:45-60

## F2: ...

## 廃止（onboarding 時点で既に無効化されていた機能）

### F-archived-1: [旧機能名]

- **廃止**: 不明（onboarding 時点でコード削除済み、git log sha1:xyz789 で削除確認）
- **痕跡**: README.md に言及あり、実装なし
- 以降復活する場合は復活条件適用必須
```

---

## F 番号採番ルール

1. **連番**: F1 から順に、抽出順ではなく依存順で採番
2. **基幹機能優先**: 他の F 番号から呼ばれている機能を先に採番
3. **抽出不能機能**: F-extract-N（採番しても仕様が曖昧なもの）として区別
4. **廃止済み機能**: F-archived-N として別採番（現行 F 番号と衝突しない）
5. **番号変更禁止**: 一度採番したら後続サイクルで変更しない（INTENT.md の追跡性維持）

---

## 確度メタデータの 3 段階

| 確度 | 意味 | 付与条件 |
|---|---|---|
| **コード確認** | ソースコード本体から直接読み取れた | 実装ロジックが明瞭、コメント・テストあり |
| **git 履歴確認** | git log / PR / blame から意図を確認 | 実装単体では不明だが履歴で補完 |
| **AI 推定** | AI が文脈から推定 | 履歴・コメントなし、推論のみ |

後続 spec-architect 人間レビューで **AI 推定 → 人間確定** へ昇格させる。

---

## to-be セクションの扱い

- onboarding では **to-be を実装しない**
- to-be は spec-architect への引き継ぎ材料として SPEC.md に残す
- spec-architect 起動後、人間レビューで以下を判定:
  - 採用 → SPEC.md の as-is 側に F 番号追加で組み込む
  - 却下 → INTENT.md の「却下案」リストに記録（再提案防止）
  - 保留 → SPEC.md の to-be セクションに残す

---

## 記述ルール

- 実装パスは必ず `file_path:start-end` 形式で記述
- 確度メタデータは省略禁止。不明なら `AI 推定 (低確度)` と明記
- to-be 提案は「発見内容」「提案」「根拠」「影響範囲」4 項目を埋める
- 機密領域は内容を書かない。存在のみ記録
- 凍結線との連動は F 番号ごとに `凍結線:` 行で明示

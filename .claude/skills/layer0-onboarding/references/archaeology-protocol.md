# Archaeology プロトコル

既存プロジェクトから事実を抽出する「考古学的」手順。
onboarding スキルの処理フロー 2 で実行される。

---

## 深度制御（4 レベル）

アーキテクチャ層（L0/L1/L2）との記号衝突を避けるため、深度は英語語彙で表現する。

| 深度 | 抽出対象 | 所要時間目安 | 典型ユースケース |
|---|---|---|---|
| **shallow** | README.md / package.json / 主要 entry point | 30 分〜1 時間 | プロトタイプ・個人プロジェクト・素性が明確 |
| **standard**（デフォルト） | shallow + 全ソースコード概観 + 過去 3 ヶ月の git log | 2〜4 時間 | 中小規模・単一ドメイン・標準的レガシー |
| **deep** | standard + 全 git log + blame + CI 設定 + 旧 wiki | 0.5〜1 日 | 複数担当者が触った・履歴重要・コンプラ系 |
| **full** | deep + 全 PR コメント + 外部連携先確認 + 運用ログ | 1〜3 日 | 長期運用・ミッションクリティカル・組織継承 |

### デフォルトは standard

人間指示がない場合は standard で開始。抽出中に以下を検出したら上位深度への昇格を提案する。

- 主要機能のコードが git log と乖離（ghost commit の痕跡） → deep 昇格提案
- 外部 API / 認証情報の使用痕跡 → deep 昇格提案
- 未ドキュメント化の運用手順が複数発見 → full 昇格提案

---

## 対話深度判定

Archaeology 開始時に人間へ提示する質問（深度合意）:

1. 「このプロジェクトは何年稼働しているか」
2. 「何人の開発者が触ったか」
3. 「ドキュメントはどの程度揃っているか（なし / README のみ / docs/ あり / wiki 豊富）」
4. 「onboarding にかけられる人間の時間はどの程度か」

回答から深度を AI が提案し、人間が承認する。曖昧応答なら standard で開始。

## 抽出フォーマット

### 機能抽出

| 項目 | 記述内容 |
|---|---|
| F 番号 | 仮採番（後続の reverse-spec で確定） |
| 機能名 | AI が命名（人間承認で最終確定） |
| 実装パス | `src/xxx.ts:123-456` 形式 |
| 呼び出し元 | 参照している他機能・外部呼び出し |
| 前提条件 | コードから読み取れた事前条件 |
| 事後条件 | コードから読み取れた事後状態 |
| 確度 | **コード確認** / **git 履歴確認** / **AI 推定** |
| 抽出不能な部分 | 推定できなかった領域（DONT.md に移送候補） |

### 意図抽出（INTENT.md への展開素材）

| 項目 | 記述内容 |
|---|---|
| 意図 | git log / PR コメント / README から読み取れた「なぜ」 |
| 条件 | 実装から逆算した条件 |
| 優先度 | 利用頻度・依存数から推定 |
| 確度 | 上記と同じ 3 段階 |

## 機密領域の扱い

抽出中に以下を検出した場合、**内容は抜かず存在のみを記録**する。

- `.env` / `secrets/` / `credentials.*` / 秘密鍵 / API キー
- 個人情報を含むテストデータ
- 社内固有 URL / 認証エンドポイント

記録フォーマット:

```markdown
## 機密領域（存在のみ記録）

- `/.env.production` — 運用環境変数（内容は抽出対象外）
- `/config/oauth-secrets.yaml` — OAuth 秘密情報（内容は抽出対象外）
```

該当ファイルは SPEC.md / INTENT.md の参照には使うが、内容を引用しない。

---

## 抽出不能領域の扱い

以下に該当する領域は DONT.md の「抽出不能」セクションへ移送する。

- コードのみでは意図を推定不能（例: マジックナンバーの由来）
- 外部依存で内部が見えない（第三者 API の挙動）
- 過去開発者の暗黙知（git log / ドキュメントに残っていない）

記録フォーマット:

```markdown
## 抽出不能（DONT.md 移送）

- F7 タイムアウト値 3000ms の根拠: 不明（2023 年のコミット以降変更なし、コメント・PR 記載なし）
- 外部 API X の応答スキーマ: 実レスポンスのみ確認、仕様書到達せず
```

spec-architect 引き継ぎ時に、人間に「この部分は現状凍結でよいか」を確認する判断材料となる。

---

## 抽出の停止条件

以下のいずれかで Archaeology を停止し、人間承認フェーズへ進む。

- 指定深度（shallow/standard/deep/full）の抽出対象を全て走査完了
- 抽出結果のサイズが SPEC.md 想定サイズの 3 倍を超えた（過剰抽出の兆候）
- 人間が途中で「ここまででよい」と指示した
- 抽出不能領域が全体の 30% を超えた（それ以上掘っても精度向上が見込めない）

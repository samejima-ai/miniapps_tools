# 12 軸の合格基準と判定例

本ドキュメントは crosscut-issue-quality-gate の 12 軸について、各軸の詳細な合格基準・判定例・層別マトリクスを定義する。

## 軸別詳細

### 軸 A: 仕様整合

**機械検査**:
- SPEC.md / DONT.md の grep 検査
- プロジェクト固有禁止事項の違反検出
- 既存 ADR との矛盾検出

**AI 推論**:
- Issue の意図が SPEC.md の方針と意味的に整合するか
- DONT.md で禁止されている設計パターンを含んでいないか
- 既存アーキテクチャとの一貫性

**合格基準**:
- SPEC.md の記述と矛盾しない
- DONT.md の禁止事項に違反しない
- 既存設計方針との整合性が取れている

### 軸 B: データ整合

**機械検査**:
- schema diff の検出
- migration script の必須化チェック
- データベース制約の変更検出

**AI 推論**:
- 後方互換性への影響評価
- データ移行戦略の妥当性判定
- schema-evolution.md との整合性

**合格基準**:
- schema-evolution.md の規約に準拠
- 破壊的変更時は移行計画が明示されている
- 後方互換性が考慮されている

### 軸 C: コード整合

**機械検査**:
- 既存抽象の grep 検出（同名 export の検出）
- 重複する機能の実装検出
- import/export パターンの一貫性

**AI 推論**:
- 新規抽象作成の必要性判定
- 既存抽象の再利用可能性評価
- 抽象レベルの適切性

**合格基準**:
- 既存抽象を適切に再利用している
- 新規抽象作成時は理由が明示されている
- コードの重複が最小化されている

### 軸 D: 依存整合

**機械検査**:
- package.json の diff チェック
- licence-checker による ライセンス検証
- bundle size の増加量測定

**AI 推論**:
- 依存関係のメンテナンス状況判定
- セキュリティ脆弱性の評価
- 代替ライブラリの検討

**合格基準**:
- ライセンスが適切（GPL コンフリクト等なし）
- bundle 増加が 10% 以下
- 最終 commit が 12ヶ月以内

### 軸 E: 品質均一化

**機械検査**:
- テスト必須項目の grep 検査
- sensor 配置規格の確認
- カバレッジ閾値の維持

**AI 推論**:
- テスト粒度の妥当性評価
- 品質基準との整合性判定
- テスト戦略の適切性

**合格基準**:
- 既存と同等以上のテスト/sensor 規格
- テスト計画が明示されている
- 品質基準を満たしている

### 軸 ii: 重複検出

**機械検査**:
- 既存 Issue title/body の n-gram 類似度計算
- キーワード重複の検出
- 同一作業の識別

**AI 推論**:
- 意味的な重複判定
- 目的・手段の類似性評価
- 差分の明示的記述確認

**合格基準**:
- 類似度 >0.7 の既存 Issue が無い
- または明確な差分が明示されている
- 重複作業の回避が確認されている

### 軸 iii: セキュリティ

**機械検査**:
- secrets scan の実行
- auth 関連 grep の実行
- セキュリティパターンの検出

**AI 推論**:
- 認可モデルとの整合性判定
- OWASP top 10 観点での評価
- セキュリティリスクの評価

**合格基準**:
- OWASP top 10 観点でリスクなし
- 認可表との整合が取れている
- セキュリティベストプラクティスに準拠

### 軸 v: 後方互換

**機械検査**:
- API schema diff の検出
- DB schema diff の検出
- semver の必須化チェック

**AI 推論**:
- 影響範囲の判定
- ロールバック可能性の評価
- 移行戦略の妥当性

**合格基準**:
- 破壊的変更時はロールバック計画必須
- semver 規約に準拠
- 影響範囲が明確化されている

### 軸 vi: 観測性統一

**機械検査**:
- 収集層: schema 統一確認、命名規則 grep
- 提示層: dashboard 項目数チェック、カーディナリティ閾値確認

**AI 推論**:
- 収集層と提示層の二層整合性判定
- Council 入力データとしての適格性評価
- 人間可読性の評価

**合格基準**:
- 収集層 = 精緻・構造化、薄くしない
- 提示層 = P3 が一目で解釈可能
- 両層で命名規則統一、drilldown 可能

### 軸 vii: ドキュメント波及

**機械検査**:
- INDEX.md/README/ADR 更新の grep 確認
- 関連ドキュメントの網羅性チェック
- リンク整合性の確認

**AI 推論**:
- ドキュメント更新の必要性判定
- 情報の一貫性評価
- 利用者視点での理解可能性

**合格基準**:
- 該当ドキュメントへの更新が含まれている
- 情報の一貫性が保たれている
- 必要な説明が追加されている

### 軸 viii: テスト粒度

**機械検査**:
- テストファイルの存在確認
- カバレッジ閾値のチェック
- テスト命名規約の確認

**AI 推論**:
- TDD 妥当性の評価
- テスト計画の適切性判定
- テストケースの網羅性評価

**合格基準**:
- Issue 段階で test-first 計画が明示されている
- テストカバレッジが基準を満たしている
- 適切なテスト粒度が設定されている

### 軸 ix: 並列安全性

**機械検査**:
- scope/mutex/depends-on ラベル grep
- in-progress Issue list 取得
- ラベル整合性チェック

**AI 推論**:
- 影響範囲衝突の意味的判定
- 並列実行リスクの評価
- 依存関係の妥当性判定

**合格基準**:
- scope/mutex ラベルが適切に設定されている
- depends-on に循環依存がない
- in-progress Issue と影響範囲衝突がない

## 層×軸マトリクス

各軸の焦点は層（FE / BE / インフラ）ごとに変わる。「軸定義は層不変・焦点は層依存」の原則に基づく：

| 軸 | FE 焦点 | BE 焦点 | インフラ焦点 |
|---|---|---|---|
| A | UI/UX 仕様整合 | API 仕様整合 | デプロイ仕様整合 |
| B | 状態管理 schema | DB schema | 設定管理 schema |
| C | コンポーネント抽象 | ビジネスロジック抽象 | インフラ抽象 |
| D | npm 依存 | サーバー依存 | システム依存 |
| E | FE テスト規格 | BE テスト規格 | インフラテスト規格 |
| ii | UI 重複 | 機能重複 | リソース重複 |
| iii | XSS/CSRF 対策 | 認証・認可 | ネットワークセキュリティ |
| v | UI 後方互換 | API 後方互換 | デプロイ後方互換 |
| vi | UI 観測性 | アプリケーション観測性 | システム観測性 |
| vii | UI ドキュメント | API ドキュメント | 運用ドキュメント |
| viii | コンポーネントテスト | ユニット・統合テスト | インフラテスト |
| ix | FE 並列開発 | BE 並列開発 | インフラ並列変更 |

## 判定例

### 合格例

**軸 A（仕様整合）**:
```
✅ Issue: "ユーザー認証機能の追加"
- SPEC.md に認証方式の記述がある
- DONT.md でパスワード平文保存が禁止されており、それに準拠した実装計画
- 既存セキュリティ要件との整合が取れている
```

**軸 vi（観測性統一）**:
```
✅ Issue: "注文処理のメトリクス追加"
収集層:
- structured logging で JSON 形式
- metric 名: order_process_duration_seconds（既存命名規則準拠）
- trace ID を全 span で保持

提示層:
- ダッシュボード項目: 注文成功率・平均処理時間・エラー率（3 項目）
- P3 が一目で判断可能な粒度
- 収集層の metric から drilldown 可能
```

### 不合格例

**軸 A（仕様整合）**:
```
❌ Issue: "新しい UI ライブラリの導入"
- SPEC.md で既存 UI フレームワークの統一が明記されている
- DONT.md で UI ライブラリの無秩序な追加が禁止されている
- 理由ラベル: gate-failed:axis-A
```

**軸 ix（並列安全性）**:
```
❌ Issue: "認証モジュールのリファクタ"
- 既に Issue #123 が `scope:auth` で in-progress
- 同一領域での並列作業でコンフリクトリスク
- 理由ラベル: gate-failed:axis-ix
```

## プロジェクト固有カスタマイズ

各プロジェクトは `assets/issue-quality-checklist.template.md` をカスタマイズし、プロジェクト固有の合格基準を設定できる：

- SPEC.md の特定セクションとの整合チェック
- プロジェクト固有のテスト規格
- 使用ライブラリの制約
- セキュリティ要件の特殊性
- 観測性基準の調整

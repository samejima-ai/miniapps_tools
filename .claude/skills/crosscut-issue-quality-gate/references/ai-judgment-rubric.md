# AI 推論 Rubric

本ドキュメントは crosscut-issue-quality-gate の各軸における AI 推論部分の判定基準・評価手順・回答形式を定義する。

## 総合方針

### 推論原則

1. **機械検査結果を前提とする**: 機械検査の結果を踏まえた上で、意味的・文脈的な判定を行う
2. **プロジェクト文脈を考慮**: SPEC.md、DONT.md、既存コードベース、過去の Issue 履歴を文脈として活用
3. **保守的判定**: 不明確な場合は不合格寄りに判定し、人間の判断を仰ぐ
4. **理由明示**: 判定結果には必ず具体的な理由を併記

### 回答形式

各軸の AI 推論結果は以下の JSON 形式で出力する：

```json
{
  "axis": "A",
  "judgment": "pass|fail|warn", 
  "confidence": 0.85,
  "reasoning": "具体的な判定理由",
  "evidence": ["根拠1", "根拠2"],
  "recommendations": ["改善提案1", "改善提案2"]
}
```

- `judgment`: pass（合格）/ fail（不合格）/ warn（要注意、人間判断推奨）
- `confidence`: 0.0-1.0 の確信度
- `reasoning`: 判定理由の要約（100文字以内）
- `evidence`: 判定根拠の具体例（機械検査結果、コード例等）
- `recommendations`: 不合格時の改善提案

## 軸別判定基準

### 軸 A: 仕様整合

**判定観点**:
- Issue の目的が SPEC.md の方針と意味的に合致しているか
- 提案する解決手段が DONT.md の禁止事項を回避しているか
- 既存アーキテクチャとの概念的一貫性があるか

**pass 基準**:
- SPEC.md の記述と方向性が一致
- DONT.md の禁止パターンを含まない
- 既存設計との矛盾がない

**fail 基準**:
- SPEC.md と明確に矛盾する方向性
- DONT.md で明示的に禁止されているアプローチ
- 既存アーキテクチャとの根本的矛盾

**warn 基準**:
- SPEC.md で言及されていない新しい領域
- DONT.md のグレーゾーン
- 既存アーキテクチャへの大きな変更を伴う

### 軸 B: データ整合

**判定観点**:
- データ変更の後方互換性への意図的配慮
- schema-evolution.md の規約との整合性
- データ移行戦略の現実的妥当性

**pass 基準**:
- 後方互換性が保たれている、または適切な移行計画がある
- schema-evolution.md の手順に準拠
- データ移行のリスクが適切に評価されている

**fail 基準**:
- 破壊的変更で移行計画が不在
- schema-evolution.md の規約を無視
- データロスのリスクが高い変更

**warn 基準**:
- 移行計画はあるが複雑でリスクを伴う
- schema-evolution.md にない新しいパターン
- 影響範囲が不明確

### 軸 C: コード整合

**判定観点**:
- 新規抽象作成の妥当性（既存抽象では対応不可能か）
- 重複実装の回避努力
- 抽象レベルの適切性

**pass 基準**:
- 既存抽象の適切な再利用
- 新規抽象作成時は明確な理由がある
- 適切な抽象レベルでの設計

**fail 基準**:
- 既存抽象を無視した重複実装
- 不要な新規抽象の作成
- 抽象レベルの不整合

**warn 基準**:
- 既存抽象の拡張が必要で影響が大きい
- 新規抽象の必要性は理解できるが設計に課題
- 抽象レベルの調整が必要

### 軸 D: 依存整合

**判定観点**:
- 新規依存のメンテナンス性・安全性
- ライセンス互換性の確認
- パフォーマンスへの影響評価

**pass 基準**:
- アクティブにメンテナンスされている依存
- ライセンスに問題なし
- パフォーマンスへの影響が許容範囲

**fail 基準**:
- 長期間メンテナンスされていない依存
- GPL 等のライセンスコンフリクト
- 大幅なパフォーマンス劣化

**warn 基準**:
- メンテナンス状況が不透明
- ライセンスは問題ないが条件付き
- パフォーマンス影響が境界線上

### 軸 E: 品質均一化

**判定観点**:
- テスト粒度・カバレッジの妥当性
- 既存品質基準との整合性
- 品質向上への貢献

**pass 基準**:
- 適切なテスト粒度と十分なカバレッジ
- 既存基準を満たすまたは上回る
- 全体的な品質向上に寄与

**fail 基準**:
- テストが不十分またはカバレッジが低い
- 既存基準を下回る品質
- 品質劣化を招く変更

**warn 基準**:
- テスト戦略は妥当だが実装に課題
- 既存基準ギリギリ
- 短期的品質低下だが長期的には改善

### 軸 ii: 重複検出

**判定観点**:
- 既存 Issue との意味的重複の有無
- 目的・手段の類似性
- 差分の明確性

**pass 基準**:
- 既存 Issue との明確な差分がある
- 目的または手段が異なる
- 重複しても明確な理由がある

**fail 基準**:
- 既存 Issue とほぼ同一の内容
- 目的も手段も重複している
- 差分が説明されていない

**warn 基準**:
- 部分的に重複しているが差分もある
- 既存 Issue の拡張として妥当か判断が困難
- 関連性が高いが独立性もある

### 軸 iii: セキュリティ

**判定観点**:
- OWASP Top 10 観点での脅威分析
- 認可モデルとの整合性
- セキュリティベストプラクティスの準拠

**pass 基準**:
- セキュリティリスクが適切に評価・対処されている
- 認可モデルとの整合が取れている
- ベストプラクティスに準拠

**fail 基準**:
- 明確なセキュリティリスクがある
- 認可モデルとの矛盾
- セキュリティ対策が不十分

**warn 基準**:
- セキュリティリスクは低いが要注意
- 認可モデルのグレーゾーン
- 新しいセキュリティ課題への対応

### 軸 v: 後方互換

**判定観点**:
- 破壊的変更の影響範囲評価
- ロールバック可能性の確認
- 移行戦略の現実性

**pass 基準**:
- 破壊的変更なしまたは適切な移行計画
- ロールバックが可能
- 移行戦略が現実的

**fail 基準**:
- 破壊的変更で移行計画なし
- ロールバックが不可能
- 移行戦略が非現実的

**warn 基準**:
- 移行計画はあるが複雑
- ロールバックは技術的に可能だが困難
- 移行戦略に不確実性がある

### 軸 vi: 観測性統一

**判定観点**:
- 収集層の精緻性・構造化度
- 提示層の人間可読性
- Council 入力データとしての適格性

**pass 基準**:
- 収集層：精緻で構造化され、薄くしていない
- 提示層：P3 が一目で理解可能
- 両層で命名規則が統一されている

**fail 基準**:
- 収集層：データが薄いまたは非構造化
- 提示層：人間が理解困難
- 命名規則がバラバラ

**warn 基準**:
- どちらかの層に改善余地がある
- 命名規則は統一されているが最適ではない
- Council 入力としては使えるが品質に課題

### 軸 vii: ドキュメント波及

**判定観点**:
- 必要なドキュメント更新の網羅性
- 情報の一貫性
- 利用者視点での理解可能性

**pass 基準**:
- 必要なドキュメントがすべて更新されている
- 情報に矛盾がない
- 利用者が理解できる説明

**fail 基準**:
- 重要なドキュメント更新が欠落
- 情報に矛盾がある
- 説明が不十分で理解困難

**warn 基準**:
- 一部ドキュメント更新が不完全
- 軽微な情報の不整合
- 説明は妥当だが改善余地がある

### 軸 viii: テスト粒度

**判定観点**:
- test-first 計画の妥当性
- テスト粒度の適切性
- TDD 原則への準拠

**pass 基準**:
- test-first 計画が明示され妥当
- 適切なテスト粒度
- TDD 原則に準拠

**fail 基準**:
- test-first 計画が不在または不適切
- テスト粒度が不適切
- TDD 原則を無視

**warn 基準**:
- test-first 計画はあるが改善余地
- テスト粒度は妥当だが最適ではない
- TDD 準拠だが一部に課題

### 軸 ix: 並列安全性

**判定観点**:
- in-progress Issue との意味的衝突評価
- 影響範囲の適切な設定
- 依存関係の妥当性

**pass 基準**:
- 既存作業との衝突がない
- 影響範囲が適切に設定されている
- 依存関係に循環がない

**fail 基準**:
- 明確な衝突がある
- 影響範囲の設定が不適切
- 循環依存が発生

**warn 基準**:
- 部分的に衝突する可能性
- 影響範囲の設定に改善余地
- 依存関係が複雑

## 特殊パターンの判定

### DH メタ Issue の場合

DH 自身に関する Issue（dh-upgrades 派生等）は以下の追加観点で判定：

1. **フラクタル原則への準拠**: DH の自己相似性を損なわないか
2. **メタ循環の回避**: 自己言及による論理的矛盾がないか
3. **独立性の保持**: harness-verifier 等の独立性を損なわないか

### 緊急パッチの場合

緊急性が高い Issue（security fix 等）は以下の緩和規定を適用：

1. **軸 viii（テスト粒度）**: 事後テスト追加を許容
2. **軸 vii（ドキュメント波及）**: 事後ドキュメント更新を許容
3. **その他の軸**: 通常基準を維持

### 実験的機能の場合

実験・PoC・spike 系の Issue は以下の特別基準：

1. **軸 v（後方互換）**: feature flag による分離を前提に緩和
2. **軸 E（品質均一化）**: 実験レベルのテストで許容
3. **軸 vi（観測性統一）**: 最小限の観測性で許容

## AI モデル固有の注意点

### Context 長制限への対応

1. **優先順位付け**: 軸 A（仕様整合）> 軸 iii（セキュリティ）> その他
2. **要約技術**: 長文 Issue は重要部分を抽出して判定
3. **分割処理**: 必要に応じて軸を分割して処理

### Hallucination 対策

1. **根拠の明示**: 判定根拠を必ず具体例と共に記載
2. **不確実性の表明**: confidence が低い場合は warn 判定
3. **機械検査結果の重視**: AI 推論単独ではなく機械検査結果を重視

### バイアス軽減

1. **多角的評価**: 複数の観点から評価
2. **反対意見の考慮**: fail 判定時は pass の可能性も検討
3. **過去事例の参照**: 類似 Issue での判定事例を参考

## 判定結果の活用

### 合格時の処理

- `ready-for-ai` ラベル付与を許可
- 判定ログを保存（将来の改善参考）
- confidence が低い場合は人間レビューを推奨

### 不合格時の処理

- 対応する `gate-failed:axis-X` ラベルを付与
- Issue に判定理由をコメント
- 改善提案を併記

### 要注意時の処理

- `needs-review` ラベルを付与
- 人間判断を要請
- AI 判定理由を詳細に記録

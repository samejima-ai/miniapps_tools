# サブエージェント統括プロトコル

L2 配下でサブエージェントを統括する際の情報純度プロトコル。
哲学的背景は `../../layer0-spec-architect/references/philosophy.md` §3（情報純度原則）を参照。

---

## 情報純度の定義

**独立した責務エージェントが受け取るコンテキストの、当該責務にとっての信号対雑音比**。

純度が低い（雑音が多い）と以下が発生する：
- 誤検出（ノイズを信号と誤判定）
- 取りこぼし（信号がノイズに埋もれる）
- 過剰修正（無関係な文脈に引きずられた判断）

純度を保つには、**責務を単一化**し、**入出力をドキュメント契約化**し、**コンテキストを独立**させる。

---

## L1 Main の責務境界

### L1 Main は実装専念

L1 Main（layer1-autonomous-dev）の責務は **AI自律駆動開発の実行** のみ。

- 仕様策定 → しない（L0 の責務）
- サブエージェント統括 → しない（L2 の責務）
- 仕様改訂 → しない（L0 にタイプC献上で戻す）

### L1 が呼ぶサブエージェント

L1 が呼ぶサブエージェントは以下に限る（情報純度4基準を全て満たすもの）：

- **layer1-independent-reviewer**（M2 以上で必須）: 実装コンテキストから隔離された独立検証

これ以外のサブエージェント起動は L1 の責務外。必要なら L0 に差し戻して L2 発動を要求する。

---

## L2 統括者の責務

### L2 は統括実行に全集中

L2 統括者（layer2-orchestrator）の責務は**与えられた配下構成でマルチエージェント統括を実行する**こと。

- 何を統括するか（配下 Agent 構成） → L0 が決める
- どう統括するか（実行順・並列化・差戻し判断） → L2 の責務
- 個別 Agent の内部処理 → 各 Agent の責務

### 配下構成の変更は L2 の責務ではない

実行中に配下構成の不足・過剰を検知した場合、L2 は**独断で変更しない**。L0 に差し戻し、REGIME.md / INTEGRATION.md の更新を要求する。

これにより：
- L0 が「何を統括するか」の責務を独占的に保持
- L2 が実行中に判断分岐を増やさない（情報純度維持）
- 設計判断の履歴が REGIME.md に集約される

---

## Agent 追加判定基準（4基準）

新規サブエージェントを立てる判断は、以下 **全て** を満たす場合に限る：

### 1. 純度向上

分離することで**受け手のコンテキスト純度が明確に上がる**。

- 既存 Agent と同じコンテキストで済むなら、分離しない
- 「分離すると管理しやすい」は理由にならない（管理コストは情報損失コストより小さい前提）

### 2. 責務明確

分離後の責務が**単一かつ独立**。

- 複数責務を兼ねる Agent は分離候補ではない（まず責務を分解してから検討）
- 他 Agent の責務と重複しない

### 3. 入出力規格化

**ドキュメント化された契約**で通信できる。

- 入力: 参照する規格ファイル（SPEC.md / sensors/* / delivery/*.md 等）
- 出力: 生成する規格ファイル（VERIFICATION.md / INTEGRATION.md / plan.md 等）
- 自然言語経由の暗黙契約は不可（情報損失前提）

### 4. 独立コンテキスト必然性

**同一コンテキストでは認知バイアスが生じる**必然性がある。

- 実装者と独立レビュアーの関係（実装者は自分の実装を甘く評価しがち）
- Planner と Generator の関係（設計意図と実装詳細の混同を防ぐ）

4基準のどれかを欠く分離は情報損失コストに見合わない。**Agent を増やしすぎない**。

## 配下Agent 構成の決定責務

### 決定は L0

L2 配下にどの Agent を起動するかは **L0 が認識擦り合せで決定**する。

- L0 が対話で複雑性の内実（E2E シナリオ数・ドメイン分割数・統合検証範囲等）を把握
- L0 が REGIME.md / INTEGRATION.md に配下構成を明示
- L2 は明示された構成を受け取って実行する

### L2 は実行に全集中

L2 は構成変更を独断で行わない。構成が不足・過剰と感じたら L0 に差し戻す。これにより：

- フラクタル原則（各責務は単一）が保たれる
- 設計判断の履歴が REGIME.md に集約される
- L2 の実行時判断分岐が増えず情報純度が保たれる

### 構成例

REGIME.md / INTEGRATION.md に記載される配下構成の例（L0 が決定）：

```yaml
L2-subagents:
  playwright-test-agents:
    enabled: true
    agents: [Planner, Generator, Healer, Reviewer]
  l1-parallel:
    enabled: true
    domains: [auth, billing, notification]
  integration-verifier:
    enabled: true
    scope: [cross-domain-invariants, e2e-scenarios]
```

L2 はこの指定を受け取り、指定通りに起動する。

### 構成変更プロトコル

L2 が実行中に「この構成では目的達成できない」と判定した場合：

1. 実行を一時停止
2. `delivery/L2-CONFIG-FEEDBACK.md` に判定根拠と変更提案を記載
3. L0 に差し戻し（タイプC献上相当）
4. L0 が REGIME.md / INTEGRATION.md を更新
5. L2 が更新された構成で再起動

## 既存 skill 構成との対応

本プロトコルは既存の L2 系 skill と以下のように対応する。

### layer2-orchestrator

- 本プロトコルの「L2 統括者」を実装する skill
- SKILL.md は発動条件・処理フロー・入出力を定義
- 本ファイル（sub-agent-protocol.md）は原則書として参照される

### layer2-integration-verifier

- 本プロトコルの「跨ぎドメイン検証」を担当する常駐 Agent
- L2 発動時に常時必須起動（REGIME.md / INTEGRATION.md で scope を L0 が指定）
- layer1-independent-reviewer と同じ骨格（情報純度4基準を満たす）

### layer1-independent-reviewer

- L1 の実装コンテキストから隔離された独立検証 Agent
- M2 以上で常時必須起動
- L2 発動時も各 L1 配下で起動される（フラクタル適用）

## 自然言語通信の情報損失

エージェント間通信は自然言語経由で必ず情報損失が発生する。これを前提にコスト計算する：

- 同一コンテキスト内のやり取りは損失ゼロ
- Agent 分離すると損失が発生するが、純度向上の効果と相殺
- 損失 < 純度向上 なら分離する。逆なら分離しない

### ドキュメント契約の効用

自然言語の直接通信ではなく、**ドキュメント化された規格ファイル経由**で通信することで：

- 情報損失を最小化（規格が曖昧性を排除）
- 非同期実行が可能（待機・再開時にもコンテキスト復元可能）
- 履歴が残る（事後評価で検証可能）

本プロトコルで規定する全 Agent 間通信はドキュメント契約経由で行う。

## 本プロトコルの適用範囲

- L2 発動時のサブエージェント統括全般
- M1/M2 での layer1-independent-reviewer 起動時（フラクタル適用）
- 将来追加される L2 配下 Agent 群（Playwright Test Agents 等）

本ファイルは L2 の原則書として機能する。個別 Agent の実装詳細は各 Agent の SKILL.md / references/ を参照。

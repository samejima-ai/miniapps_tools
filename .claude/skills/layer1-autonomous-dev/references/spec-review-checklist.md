# 仕様レビューチェックリスト

Layer 1がドキュメント受領時に実行する仕様レビューの詳細チェック項目。

---

## 実装可能性チェック

- [ ] 各機能の条件が「機能×条件」の最低粒度を満たしているか
  - 「適切に処理する」「いい感じに」等の曖昧表現がないか
  - 条件に具体的な数値・挙動が記述されているか
- [ ] 必要な外部サービス・APIが特定されているか
  - DONT.mdの「未知の外部API統合」に抵触しないか
- [ ] データモデルの概念レベルの関係が読み取れるか
  - エンティティ間の関係（1:N、N:N等）が推測可能か

## 矛盾チェック

- [ ] 機能間で矛盾する条件がないか
  - 例：機能Aでは「ユーザーは1アカウントのみ」、機能Bでは「複数アカウント切替」
- [ ] 制約と機能が矛盾しないか
  - 例：「外部API不使用」の制約と「決済機能」の両立
- [ ] 優先順位間の整合性があるか
  - 高優先の機能が低優先の機能に依存していないか

## 不足チェック

- [ ] INDEX.mdに記載された全機能がSPEC.mdで定義されているか
- [ ] 認証・認可が必要な場合、権限の定義があるか
  - 最低限「誰が何をできるか」が記述されているか
- [ ] エラー時の挙動が定義されているか
  - 全機能に対して不要。主要な操作パスのエラーハンドリングのみ

## スコープ侵食チェック

- [ ] DONT.mdに記載された領域に踏み込む要件が含まれていないか
- [ ] 創造的UXデザインを暗示する要件がないか
  - 「革新的なUI」「今までにない体験」等
- [ ] パフォーマンス最適化を前提とした要件がないか
  - 「1万同時接続」「リアルタイム処理」等、最適化が必須な要件

## 過去履歴整合性チェック（LC ≥ 1 で実行）

`history/INTENT.md` と `history/CHANGELOG.md` を読み込み、今回の SPEC と照合する。

- [ ] 過去 INTENT の条件と今回 SPEC の条件が矛盾していないか
  - 例：F001「上限30品」を維持しているのに、新 SPEC で「上限なし」になっていないか
- [ ] 廃止された機能が SPEC に復活していないか
  - `history/INTENT.md` の `**廃止**` マーカー付き項目と照合
  - 復活している場合、REGIME.md の「廃止判断プロトコル」で確認された復活条件を満たすか
- [ ] 却下案リストに載っている案が SPEC に再提案されていないか
  - 却下理由が解消されているなら可、解消されていないなら仕様レビュー結果で報告
- [ ] 確度「AI推定」の INTENT 項目が SPEC に反映されているか
  - 前回ループで AI 推定で埋めた項目を今回の SPEC で確定できているか

### 儀式スキップ（E2）時の追加責務

L0 で儀式が E2 拒否でスキップされた場合、本チェックは L1 実装**後**にも再実行する。
- 実装が過去 INTENT と矛盾していないか
- 実装が廃止機能の再現になっていないか
- 検出時は `delivery/HISTORY-DIFF.md` の「履歴整合性チェック」セクションで報告（人間に質問しない）

---

## レビュー結果の分類

| 分類 | 判定 | アクション |
|---|---|---|
| 問題なし | 全項目クリア | 開発に進む |
| 軽微な不足 | 合理的に解釈可能 | 解釈して開発に進む。フィードバックレポートに解釈内容を記載 |
| 重大な不足・矛盾 | 開発続行不可能 | 仕様レビュー結果を即献上 |

「軽微」と「重大」の境界：
- 解釈次第で機能の挙動が大きく変わるか？ → 重大
- 解釈が1通りにほぼ定まるか？ → 軽微

# WF 基底テンプレートの設計根拠

v5.3.0 で追加。`SKILL.md §原則` の「WF 形状単一性」を支える詳細根拠と運用閾値の記録。

## 背景

HANDOFF「1 機能完遂の自律駆動 WF 設計」(2026-04-30) で、bug-fix / 新規機能 / リファクタ / 仕様改訂等の機能タイプ別に WF 群を多様化するか、単一形状を保つかが論点となった。L0 (spec-architect) で 5 phase の検討を実施し、Council 合議 (`council-2026-04-30T14:50:00Z-wfbase1`) で案 H Hybrid（薄い基底 + 観測駆動の厚化）が weighted_score 4.65、案 N（多様化しない=哲学者）が 3.25 で拮抗判定。**案 H 中核 + 案 N の運用原則化**の統合採用が確定（AD-015）。

## 採用方針

### 1. 基底 WF は SKILL.md §処理フロー（1〜8）

新規 WF テンプレート（bug-fix-WF / refactor-WF / spec-revision-WF 等）は **作成しない**。
本 SKILL の §処理フロー（1〜8）が現行の「薄い基底」であり、機能タイプの違いは §4「実装タスク分解」内で context として動的に注入する。

```
処理フロー (1〜8) ←─ 単一基底（全機能タイプ共通）
   │
   └─ §4 実装タスク分解
        │
        ├─ context: bug-fix       → 修正範囲・回帰テスト・原因分析
        ├─ context: 新規機能       → 仕様充足・SPEC 突合・統合確認
        ├─ context: リファクタ     → 振る舞い不変・テスト維持・段階分割
        └─ context: 仕様改訂提案   → §3 で Type C 判定 → 実装続行 or §8 献上
```

### 2. 機能タイプ別 WF 群を作らない理由

- **philosophy.md §1 フラクタル原則 P1（同型再帰）**: 全層・全機構が形状の同型を保つことを要求する。WF を機能タイプ別に多様化すると形状の同型が壊れる。
- **職種別分業の罠との同型性**: フロント / バック / SRE 等の職種軸分業が職能のサイロ化を生むのと同型に、機能タイプ軸 WF 群は 5 年スパンで「タイプ N+1 の追加要求」を再発させる構造的可能性を持つ。
- **YAGNI 原則整合**: 早期 over-engineering を回避し、観測駆動でのみ拡張する。

### 3. 厚化（基底引き上げ）の閾値

| 観測項目 | 閾値 |
|---|---|
| 同一 override パターンの反復 | 3 機能タイプ以上で同じ override が発生 |
| 反復観測期間 | 3 ヶ月以上、または 5 cycle 以上 |
| 観測者 | L1 自身 + layer1-independent-reviewer の指摘集約 |

閾値到達時の挙動: **Council を起動して基底引き上げを諮問**。Council が承認した場合のみ §処理フローへ織り込む。L1 単独判断による基底変更は禁止（責務分離 P3）。

### 4. 観測対象外（厚化候補にしない）

以下は観測対象から除外する（基底引き上げ要求の発生源にしない）:

- **CTL レベル別の挙動分岐**: CTL は既に SK レベルで吸収されている（v5.0.0 AD-004）
- **dev_mode 別の挙動分岐**: dev_mode は既に SK レベルで吸収されている（v5.0.0 AD-002）
- **モード（M1/M2/L2）別の挙動分岐**: §モード別の実行フロー で既に吸収されている

これら 3 軸は本 §処理フローと直交する独立軸として設計済み。同一軸の重複追加は構造侵食。

## 第 3 の道（v6.0.0 候補として温存）

哲学者意見の極論として「**単一 WF + 動的 context 注入**」が提示された。これは現行の運用原則を超え、context engineering の中核に WF 設計を据え直す思想転換であり、minor 改修では実装し得ない。

v6.0.0 major 昇格時の検討候補として温存される。本 reference の本セクションが将来検討時の起点となる。

# 献上物フォーマット定義

Layer 1が人間に提出する献上物の構造と記述ルール。
哲学的背景は `../../layer0-spec-architect/references/philosophy.md` §5（献上哲学）を参照。

## 献上物タイプ一覧

| タイプ | 発生タイミング | 開発継続 | 更新対象 | 本ドキュメント内の節 |
|---|---|---|---|---|
| A: 仕様レビュー結果 | 仕様レビュー時（実装不可能判定） | 中断 | SPEC/DONT/REGIME | §タイプA |
| B: 成果物 | 開発完了時 | 完了 | 成果物一式 | §タイプB |
| C: 仕様改訂提案 | 開発中〜完了時（仕様改善余地発見） | 継続 | INTENT.md / SPEC.md（承認後） | §タイプC |
| D: 異常献上（v5.3.0 で追加） | 開発中（自力修正上限到達後の技術的解消不能） | 中断 | 環境・依存・実装手段（SPEC は更新不要） | §タイプD |

タイプA と C は似て非なる：
- タイプA: 実装**不可能**（L1 は止まる）
- タイプC: 実装**続行可能だが仕様改善余地あり**（L1 は自力判断で実装続行し、献上時に人間が確認）

タイプA と D も似て非なる：
- タイプA: **仕様起因**（SPEC/DONT/REGIME を L0 で更新する必要がある）
- タイプD: **技術例外起因**（仕様は変更不要、依存・環境・実装手段の救済を要請）

---

## 献上物タイプA：仕様レビュー結果

仕様レビューで問題を検出し、開発に入れない場合に提出する。

### 構造

```markdown
# 仕様レビュー結果

## ステータス: 開発不可

## 検出された問題

### [問題1のタイトル]
- **分類**: 不足 / 矛盾 / スコープ侵食 / 実装不可能 / 体制齟齬
- **該当箇所**: SPEC.md#[セクション名] または REGIME.md
- **詳細**: [具体的に何が問題か]
- **影響**: [この問題を放置した場合に発生する具体的な不具合]
- **提案**: [Layer 0での解決案。可能であれば選択肢を提示]

### [問題2のタイトル]
...

## 開発可能な範囲（任意）
[問題のある機能を除外すれば残りは開発可能な場合、その旨を記載]
```

### 記述ルール
- 問題は重大度の高い順に記載する
- 「問題がある」だけでなく「どうすれば解決するか」の提案を必ず添える
- Layer 0ドキュメントのどの箇所を修正すべきかを明示する
- 体制齟齬（REGIME.md のモードが実態と合わない）を検出した場合も本フォーマットで報告する

## 献上物タイプB：成果物

開発完了時に提出する。動作するコードとフィードバックレポートのセット。

### 構造

```
delivery/
├── DELIVERY.md          # 献上レポート（全モード必須、AI 内部状態 / L0 引き継ぎ用）
├── HANDOFF.md           # 人間向け引き渡しサマリー（v4.2、全モード必須）
├── VERIFICATION.md      # 独立検証レポート（M2以上で必須）
├── INTEGRATION.md       # 統合レポート（L2のみ。L2オーケストレータが生成）
├── HISTORY-DIFF.md      # 履歴層更新差分（LC ≥ 1 で必須）
├── src/                 # ソースコード
├── tests/               # テストコード
└── (その他プロジェクトファイル)
```

### delivery/ 配置規則（厳格）

`dev-env-spec.md` の「ファイル配置規則」に従い、`delivery/` 配下に置けるものは以下に限定する。

#### 許可

- **規格ファイル**: `DELIVERY.md`, `HANDOFF.md`, `VERIFICATION.md`, `INTEGRATION.md`, `HISTORY-DIFF.md`
- **成果物**: `src/`, `tests/`, ビルド生成物（`dist/`, `build/` 等）、実行に必要なアセット
- **プロジェクト固有の実行設定**: `package.json`, `requirements.txt` 等（プロジェクト側ディレクトリへの反映用）

#### 禁止

- `PLAN.md`, `PHASE*.md`, `TODO.md`, `NOTES.md`, `MEMO.md`, `WORKLOG.md` 等、L1 の作業メモ・中間計画
- L0 規格ファイル（`SPEC.md`, `DONT.md`, `REGIME.md`, `INDEX.md`, `CLAUDE.md`）の複製
- 試行錯誤の残骸（`*_old.*`, `*_backup.*`, `.bak`, `.orig` 等）
- 人間非読（自動生成キャッシュ・ログ等）

#### 違反検出時の挙動

- **自動修復可能**: 作業メモ・試行錯誤残骸の自然削除、L0 規格ファイル複製の削除 → 実行し DELIVERY.md「配置規則違反修復ログ」に記載
- **人間承認が必要**: 成果物と区別しづらいファイル（プロジェクト由来の `.bak` 等） → 修復保留し DELIVERY.md に提案記載
- **修復不可能**: 規格ファイル本体に配置違反がある（例: `DELIVERY.md` が `delivery/sub/` に置かれている） → FAIL として L1 差戻し

### DELIVERY.md テンプレート

```markdown
# 献上レポート

## ステータス: 完了 / 一部未完了

## 体制情報

- **モード**: M1 / M2 / L2
- **REGIME.md参照日**: [日付]
- **判定時のAI能力バージョン**: [REGIME.md から転記]
- **権限レベル**: L0-2 / L0-3（REGIME.md から転記、未記載時は L0-2）
- **ARC**: [monolith / realtime-pubsub / event-sourcing]（未指定時は monolith）

## 実装済み機能

- [機能A]: 完了
- [機能B]: 完了
- [機能C]: 一部完了（詳細は下記）

## 自己検証結果

### 計算的センサー
- ビルド: PASS / FAIL
- 型チェック: PASS / FAIL
- リンター: PASS / FAIL
- テスト: PASS / FAIL ([通過数]/[総数])

### 推論的センサー
- 仕様合致: PASS / FAIL
- 動作確認: PASS / FAIL
- 使用確認: PASS / FAIL
- 意図合致: PASS / FAIL / N/A（refactor-intent-map.md 不在時は N/A）

### 意図合致検証（v5.5.0 Phase γ コア 3 件、`delivery/refactor-intent-map.md` 存在時のみ追加）

`refactor-intent-map.md` 不在時は本セクション全体を**省略**する（後方互換完全維持、3 軸動作）。存在時は以下を記載：

```markdown
## 意図合致検証（archeo-architect 由来、Phase γ 本実装）

参照した意図マップ: delivery/refactor-intent-map.md (delivered_at: <timestamp>)

| Island | refactor_directive | 実装結果 | 判定根拠 | 判定 |
|---|---|---|---|---|
| Island-001 | preserve | 変更なし | 承認テスト 12/12 PASS（`delivery/approval-tests/island-001.baseline.json`） | PASS |
| Island-002 | restructure | 再構造化（意図保持） | 不一致率 0.003% < 0.01% 閾値（`delivery/reconciliation-logs/island-002.log`） | PASS |
| Island-003 | discard_and_redesign | 新規設計 | AbsentZone.redesign_directive 適合 | PASS |
| ... | ... | ... | ... | ... |

意図逸脱検出: 0 件 / N 件
```

意図逸脱検出があった場合は `failure: intent_drift` として L1 自力修正を試行する。修正不能なら **Type C 献上**（仕様改訂提案）または **Type D 献上**（技術例外）に分類する。

### 独立検証（M2以上で必須）
- 検証レポート: VERIFICATION.md参照
- 最終判定: PASS / FAIL

## 仕様改訂提案（タイプC、任意）

[開発中に発見した仕様の改善提案。Layer 0 への逆流情報]

### タイプCプロトコル

- **発生条件**: L1 が実装続行可能だが、SPEC.md の曖昧さ・改善余地を発見した場合
- **L1 の挙動**: **自力判断で実装続行**（philosophy.md §4「自律駆動を止めない」）
  - 発見のたびに L0 に戻らない
  - 合理的な解釈を採用し、DELIVERY.md の本セクションに集約する
- **人間の挙動**: DELIVERY.md 受領時に本セクションを確認
  - 承認 → INTENT.md / SPEC.md を更新（次サイクル以降に反映）
  - 却下 → L1 の解釈を仕様として追記
  - 保留 → 次回 F2 儀式で再議論

タイプA（実装不可能で中断）と区別すること。タイプCは**実装続行前提**の逆流情報。

### [フィードバック1のタイトル]
- **該当箇所**: SPEC.md#[セクション名]
- **発見内容**: [開発中に気付いた仕様の曖昧さ・改善余地]
- **AIの解釈**: [今回の実装でどう解釈したか]
- **提案**: [仕様をどう修正すると次回以降の品質が上がるか]

## 未解決事項（任意）

[自力修正を試みたが解決できなかった問題]

### [未解決事項1]
- **内容**: [何が問題か]
- **試行した対策**: [何を試したか]
- **失敗理由**: [なぜ解決できなかったか]

## 献上物タイプD：異常献上（v5.3.0 で追加）

AI が自己解決不能な技術的例外に遭遇した場合の献上物。タイプA（仕様起因）と区別される。

### タイプDプロトコル

- **発生条件**:
  - 依存破損（必要パッケージの取得不能、API 廃止、バージョン非互換 等）
  - 環境構築不能（ビルド失敗が再現性高く解消されない、必須コマンド不在 等）
  - 想定外例外（仕様上は実装可能だが実装段階で技術的に達成不能と判明 等）
  - 自力修正上限（`SKILL.md §6` の上限値）に到達後も解消しない技術的障害
- **L1 の挙動**: 自己解決を試行（自律修正上限まで）→ 解消不能と判定された時点で本献上を発する。**SPEC は変更不要**（仕様起因ではなく技術例外起因のため、L0 への差し戻しは行わない）
- **人間の挙動**: DELIVERY.md 受領時に本セクションを確認
  - 環境・依存・実装手段の救済を提供（依存差し替え、環境再構築、別実装手段の提示 等）
  - 技術的解消が困難な場合のみ SPEC 側調整を検討（その場合は人間判断で L0 を起動して Type A 経路に転換）

タイプA（仕様レビュー時に実装不可能と判定）とは経路が異なる。タイプA は **開発に入る前** に
仕様起因で発する経路、タイプD は **開発中** に技術例外起因で発する経路。

### 構造

```markdown
# 異常献上（Type D）

## ステータス: 中断（技術的解消不能）

## 例外詳細

### [例外1のタイトル]
- **種別**: 依存破損 / 環境構築不能 / 想定外例外 / その他
- **該当箇所**: [ファイルパス・行番号・コマンド等]
- **stacktrace または再現手順**: [具体的な再現方法]
- **自己解決試行履歴**:
  1. [試行1] → [結果]
  2. [試行2] → [結果]
  3. [試行N] → [結果]
- **失敗理由の推測**: [なぜ自己解決できないか]
- **人間判断期待値**: [どのような救済が必要か。例: 依存パッケージの差し替え、別実装手段の提案、環境再構築指示]
- **緊急度**: 高 / 中 / 低
```

### 記述ルール

- 自己解決試行履歴は**省略禁止**（「とりあえず Type D」の濫用を構造的に予防する）
- 試行履歴は最低でも 2 件以上（自律修正上限が 3 の前提では 3 件以上が望ましい）
- 緊急度は人間の対応優先度判定に使用（高 = ブロッカー、低 = 並行案件で吸収可能）
- SPEC 側調整が必要と L1 が判断する場合でも本献上では SPEC 修正提案を書かない
  → 人間が SPEC 起因と再判定した場合のみ Type A 経路で再起動する（責務分離）



### 体制妥当性の自己評価

- **判定モード**: M1 / M2 / L2
- **実績から見た妥当性**: 過剰 / 妥当 / 過少
- **評価根拠**: [下記チェックリスト参照]

### 評価チェックリスト

#### 過剰判定の兆候
- [ ] layer1-independent-reviewer が何も指摘しなかった（M2以上）
- [ ] ドメイン間通信がほぼ発生しなかった（L2）
- [ ] 自力修正ループが1回も発生しなかった
- [ ] 検証に要した時間が実装時間を大きく上回った

該当多数 → 次回は一段階下のモードで足りる可能性

#### 過少判定の兆候
- [ ] 自力修正ループが3回上限に達した
- [ ] 未解決事項が2件以上発生した
- [ ] 仕様レビューで軽微な不足として流したが、実装で困った
- [ ] コンテキスト窓の圧迫を感じた（M2で仕様が収まりきらない等）
- [ ] 複数の独立ドメインが暗黙に発生し、整合性取りに苦労した

該当多数 → 次回は一段階上のモードが必要（M1→M2, M2→L2）

## 事後評価の蓄積活用

DELIVERY.md の体制事後評価セクションは、複数プロジェクトを通じて蓄積することで
以下のメタ知見を得られる：

- **判定プロトコルの精度**：過剰判定率・過少判定率の把握
- **スコアリング表のキャリブレーション**：どの小項目の配点が過大／過小か
- **モード境界の再検討**：M1⇔M2 の境界閾値調整、L2発動閾値の見直し
- **AI能力バージョン別の傾向**：モデル更新でどの閾値が緩和可能になったか
- **ユーザー別の傾向**：仕様確定度を自己評価する癖のズレ補正

これらは Layer 0 の `references/regime-assessment.md` に反映し、運用知として育てていく。
特に **AI能力バージョン** 記録は将来モデルでの閾値緩和判断に必須。

## HANDOFF.md との関係（v4.2 で新設）

L1 ステップ 8「献上」では DELIVERY.md と HANDOFF.md を同時生成する。
HANDOFF.md の規格詳細は `handoff-format.md` を参照。

| ファイル | 役割 | 可読性優先 | デフォルト表示 |
|---|---|---|---|
| DELIVERY.md | AI 内部状態、L0 spec-architect への引き継ぎ | AI 可読性優先 | 要求時のみ展開 |
| HANDOFF.md | 人間向け献上物、Council 判定ダイジェスト | 人間可読性優先 | 完成時・献上時に自動表示 |

### 生成順序

1. DELIVERY.md を先に生成（L1 内部の処理結果を全て記録）
2. DELIVERY.md から人間可読部分を抽出して HANDOFF.md を生成
3. 両ファイルを献上物として提示
4. デフォルト表示は HANDOFF.md のみ

### 体制事後評価との関係

DELIVERY.md の「体制事後評価」セクションは HANDOFF.md には含めない
（エンジニア向けの内部評価のため）。要求時のみ DELIVERY.md を展開して表示する。

### 既存 history/SUMMARY.md との衝突回避

`history/SUMMARY.md`（L0 振り返り儀式エントリポイント / 圧縮履歴サマリ）と
本 HANDOFF.md は完全に別ファイル。役割も配置場所も異なる:

- `history/SUMMARY.md`: AI 用、L0 ritual 用、LC ≥ 1 で再生成
- `delivery/HANDOFF.md`: 人間用、献上時に毎回生成、LC 不問

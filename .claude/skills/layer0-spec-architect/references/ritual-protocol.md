# 振り返り儀式プロトコル

L0（spec-architect）が対話開始時に実行する「過去文脈⇔現欲求」照合プロトコル。
1→5運用で履歴層を活用し、認識ズレ・車輪の再発明・訂正漏れを防ぐ。

---

## 設計原則

- **対話コスト最小化**: 儀式は必要最小限に。無駄な質問は一切しない
- **人間は気付き役、AI は検出・提示・記録役**: 人間にできないことを AI が担う
- **曖昧応答は AI 充当**: 「どうでもいい」シグナルと解釈し、推定値で埋める
- **儀式拒否は尊重**: 人間欲求を優先し、代わりに AI が背景で照合を回す
- **形骸化防止**: 儀式自体を履歴化して効果測定する

---

## 儀式レベル（4段階）

| レベル | 条件 | 処理 | コスト目安 |
|---|---|---|---|
| 0 | `history/` なし、または前回ループから経過なし | 完全スキップ | ゼロ |
| 1 | 30日以内、規模変化なし、LC=1 | SUMMARY.md のみ読込、1問確認 | 〜30秒 |
| 2 | 30日超 or 機能追加・変更を含む対話 | SUMMARY + 関連 INTENT、2〜3問 | 〜3分 |
| 3 | LC=2 保守 or 90日超 or 規模変動±2以上 | history/ 全層読込、5〜10問 | 〜15分 |

### ベースレベル判定フロー

```
1. history/ 存在確認
   なし → レベル 0
2. LC 判定（regime-assessment.md のロジックに従う）
   LC=0 → レベル 0
   LC=2 or 最終献上から 90日超 → レベル 3
3. 対話冒頭の発話内容を軽く解析
   機能追加/変更/廃止の示唆あり → レベル 2
   バグ修正のみ → レベル 1
   規模変動 ±2 以上の示唆 → レベル 3
4. 上記で決まらない場合 → レベル 1（デフォルト）
```

---

## 動的格上げ

儀式実行中に以下を検出した場合、その場で一段階格上げ。

| 検出シグナル | 格上げ |
|---|---|
| 過去 INTENT と矛盾する欲求 | 1→2 |
| 廃止機能の復活要求 | 1→2 |
| 却下済み案の再提案 | 1→2 |
| 大規模リファクタの示唆 | 2→3 |
| 規模判定が L2 閾値接近 | 強制 3 |
| 訂正カスケード検出（1訂正が複数機能に波及） | +1 |

## 動的格下げ

以下を検出したら一段階格下げ。対話コスト最小化を優先。

| 検出シグナル | 格下げ |
|---|---|
| `history/` がほぼ空（INTENT 3件以下） | 2→1 |
| 人間が明確に「軽くやりたい」と発話 | 2→1 |
| 純粋なバグ修正のみで仕様変更なし | 1→0 |
| **曖昧応答が2回連続**（E1対応） | 現レベル −1 |

曖昧応答 2回連続は「記憶が薄い領域＝重要度低」のシグナルと解釈し、深掘りを自動停止する。

## 儀式の3フェーズ（実行時）

レベル 1 以上で実行。各フェーズはレベルに応じて深度が変わる。

### F1: 過去文脈サマリ提示（AI主導・質問なし）

1. SUMMARY.md の鮮度判定
   - `SUMMARY.md` 最終更新日 < `history/` 内他ファイル最新更新日 なら再生成
   - 一致なら使い回し（詳細は `history-layer-spec.md` のSUMMARY自動生成トリガー）
2. レベルに応じた履歴ロード
   - レベル 1: SUMMARY.md のみ
   - レベル 2: SUMMARY + 対話話題に関連する INTENT 項目（AI が関連性フィルタで選別）
   - レベル 3: SUMMARY + 全 INTENT + CHANGELOG 直近10件 + REGIME-LOG 直近5件
3. 人間に**1問だけ**提示: 「前回ここまで進みました。今回はこれの続きで合ってますか？」

### F2: 認識ズレ検出（AI主導・検出時のみ質問）

1. 新欲求 × 過去 INTENT の照合
2. 以下のパターンを AI が自動検出:
   - 矛盾: 過去の条件と両立しない欲求
   - 復活要求: 廃止機能の機能復活を示唆
   - 再提案: 却下案リストに載っている案の再提案
3. 検出時のみ人間に確認:
   - 矛盾 → 「F001 の『上限 30 品』と今の『上限なし』は両立しないです。どちらを優先？」
   - 復活要求 → 「F003 は 2026-03-20 に利用率低で廃止しました。復活条件を満たしますか？」
   - 再提案 → 「上限なし案は当初『パフォーマンス劣化リスク』で却下しました。状況変化ありますか？」
4. 検出なしなら**質問なし**でF3へ

### F3: 履歴更新予告（人間は承認のみ）

1. 対話で確定した更新内容を AI が**通知形式**で宣言
   - 「以下を INTENT.md に追加します」
   - 「F001 の条件を v2 に更新します」
   - 「R=2（他人配布）のため REGIME.md の R スコアを +1 更新します」
2. 人間が承認 or 修正指摘
3. 修正指摘ありなら該当部分のみ修正して再提示

**重要**: 履歴の実際の書き込みは L1 の献上タイミングで行う（`layer1-autonomous-dev/references/delivery-format.md` の「履歴層更新差分」セクション参照）。F3 はあくまで「予告」。

## E1対応: 曖昧応答の扱い

儀式中に曖昧応答（「たぶん」「どうだったかな」「どっちでもいい」等）が返った場合の処理。

### 基本方針

曖昧応答 = **「AI が良きに計らえ」シグナル**と解釈する。質問を重ねず、AI が最適解で充当する。

### A: AI充当 + 通知型（常時適用）

1. AI が過去 INTENT・CHANGELOG・状況データから最適解を選択
2. 「〇〇と推定しました。違ったら次回教えてください」と**通知のみ**（再質問しない）
3. 対象項目に `**確度**: AI推定 (YYYY-MM-DD)` メタデータを付与
4. SUMMARY.md の「要再確認リスト」に追加
5. 次ループの F2 で「AI 推定項目に異議ないか」を**1問にまとめて**一括確認

### C: 儀式軽量化（併用）

曖昧応答が **2回連続**発生したら、儀式レベルを自動 −1 格下げ。

- 記憶が薄い領域＝重要度低の推定として深掘りを停止
- 格下げ後も F1/F2 の検出は継続（ただしより簡略化）

---

## E2対応: 儀式拒否への対応

「儀式いらない、すぐ作って」等の拒否発話を検出した場合の処理。

### 基本方針

儀式拒否 = **「今すぐ作りたい」欲求の優先表明**。拒否は全面受容。代わりに AI が **L1 実装中に背景で履歴照合を回す**（人間にできないことを AI が担う）。

### 実行手順

1. 儀式を完全スキップ（F1/F2/F3 全て省略）
2. REGIME-LOG.md に「儀式拒否: 第N回」を記録
3. L1 起動時に以下を暗黙の追加タスクとして引き渡す:
   - 過去 INTENT との整合性チェック（実装完了後）
   - 廃止機能への回帰検出
   - 却下案の再実装検出
4. 矛盾検出時のみ `DELIVERY.md` の「履歴整合性チェック」セクションで報告（人間に質問はしない、報告のみ）

### 連続警告

拒否が **5回連続**したら、SUMMARY.md の「注意事項」セクションに以下を**常設表示**:

```
⚠ 直近5回儀式スキップ、認識ズレ蓄積の可能性
```

- 強制介入はしない（人間裁量を最終尊重）
- 警告解除条件: 1回でも儀式を実施すれば即解除

連続回数は REGIME-LOG.md の「儀式拒否」フィールドで追跡。

## 形骸化防止メカニズム

儀式自体が形骸化しないよう、以下を組み込む。

### 1. 儀式実行ログ

CHANGELOG.md の「儀式記録」セクションに毎回追記（`history-layer-spec.md` のCHANGELOGテンプレ参照）。

- レベル（動的格上げ/格下げの履歴も含む）
- スキップの有無
- 検出件数（矛盾/復活/再提案）

### 2. スキップ連続警告

儀式スキップが **5回連続**で SUMMARY.md に警告常設（E2対応と同じ閾値）。

### 3. 儀式効果の事後評価

L1 の「体制事後評価」（`delivery-format.md`）に以下を追加記録:

- 儀式で検出された件数
- 検出により回避された手戻り（推定）
- 儀式時間 vs 実装時間の比率

連続して「検出ゼロ」が続く場合は REGIME-LOG.md に警告し、次回レベル判定を一段下げ検討。

## レベル別の所要コスト目安

| レベル | 対話ターン | 所要時間 | 履歴読込量 |
|---|---|---|---|
| 0 | 0（スキップ） | 0秒 | なし |
| 1 | 最大 1 問 | 〜30秒 | SUMMARY のみ |
| 2 | 最大 3 問 | 〜3分 | SUMMARY + 関連INTENT |
| 3 | 最大 10 問 | 〜15分 | history/ 全体 |

これらは目安。動的格下げが効けば上位レベルでも短縮される。

## 儀式と L0 本来フロー（対話・モード判定）の接続

`SKILL.md` の処理フローでは、ステップ 2（対話による具体化）の**前**に儀式を実行する。

```
1. イメージ受領
1.5. 振り返り儀式（LC 判定 → レベル判定 → F1〜F3 実行）  ← 本プロトコル
2. 対話による具体化（儀式で得た過去文脈を前提に進行）
3. ドキュメント化
4. モード判定
5. 人間レビュー
6. 開発環境構築
7. 出力
```

新規プロジェクト（LC=0）では儀式レベル 0 により完全スキップ。0→1 開発には一切のオーバーヘッドが発生しない。

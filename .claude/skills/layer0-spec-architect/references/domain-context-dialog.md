# ドメイン文脈対話プロトコル

業界・業務固有の前提条件、慣習、規制、既存システム制約を Layer 0 対話で引き出し `DOMAIN-CONTEXT.md` に構造化する規格。
業務システム運用型プロジェクトで「技術は動くが業務が回らない」事故を避けるため導入。

---

## 原則

- DOMAIN-CONTEXT.md は**任意**ファイル。純粋な技術プロジェクト（汎用ツール / 実験）では不要
- 業界固有の標準パック（finance-jp.md / healthcare-jp.md 等）を**事前に作成しない**（射程外）
- 機密情報は `DOMAIN-CONTEXT.secret.md` として分離し `.gitignore` 対象にする
- DOMAIN-CONTEXT.md は Layer 0 対話で随時更新する。完成形を最初から目指さない
- フラクタル原則に従い、DOMAIN-CONTEXT.md の発見が新機能要件の発見につながる場合は SPEC.md に戻して機能化する

---

## 対話で引き出す 5 カテゴリ

### Cat-1: 業界・業務領域

**質問例**:
- どんな業界のプロジェクトか？（金融 / 医療 / 小売 / 公共 / 教育 / 製造 / その他）
- 社内向けか、顧客向けか、その両方か？
- 類似のプロダクト／競合で参考になるものはあるか？

**DOMAIN-CONTEXT.md への記録例**:
```
## 業界
- 国内物流業界（B2B）
- 荷主とトラック事業者のマッチングが主業務
- 類似: 〇〇社の △△ サービス
```

### Cat-2: 業務慣習・商習慣

**質問例**:
- この業務特有の慣習・ルールで、IT 知らない人には分かりにくいものは？
- 「紙で運用されていた時代の名残」があるか？
- 業務用語で、**他業界と同じ言葉なのに意味が違う**ものは？

**DOMAIN-CONTEXT.md への記録例**:
```
## 業務慣習
- 「〆日」は月末ではなく毎月 20 日
- 「請求書」は発行＝確定ではなく、相手承認後に確定扱い
- 「運賃」は見積・確定・請求で 3 段階で変わりうる（合意変更は電話ベース）
```

### Cat-3: 既存システム制約

**質問例**:
- 今使っているシステム／ツールは？ それを置き換えるか併用するか？
- データ連携が必須な外部システムは？（SAP / SFDC / 社内基幹 / 会計SaaS 等）
- 接続方式は？（API / CSV / 手動コピペ / FAX）

**DOMAIN-CONTEXT.md への記録例**:
```
## 既存システム
- 基幹: 社内 Oracle DB（read-only API あり）
- 会計: freee（OAuth2 API）
- 連絡: 現行は FAX＋電話。置き換え対象
```

### Cat-4: 規制・コンプライアンス

**質問例**:
- 法令・業法の準拠が必須か？（個人情報保護法 / 下請法 / 業界固有法）
- 監査対応が必要か？（年次監査 / 内部監査 / 外部認証）
- 保管期間の法定要件は？（契約書 7 年 / 税務帳簿 10 年 等）

**DOMAIN-CONTEXT.md への記録例**:
```
## 規制
- 個人情報保護法（荷主担当者の連絡先保持）
- 下請法（運賃支払期日 60 日以内の遵守必須）
- 取引記録の保管: 7 年
```

`nfr-scoring.md` の **C（コンプライアンス）スコア**と連動する。

### Cat-5: 用語集

**質問例**:
- 社内／業界でよく使われる略語・専門用語は？
- 同じものを複数の呼び方で呼んでいるものは？（正式名称 vs 通称）

**DOMAIN-CONTEXT.md への記録例**:
```
## 用語集
- 「ドライバ」= トラック運転手（IT 用語のデバイスドライバではない）
- 「バース」= 荷受け場の積降ろし待機枠
- 「着時間」= 到着予定時刻（ATD ではなく ETA）
```

## 機密情報の扱い

### 分離ルール

公開可能な文脈と機密を分ける:

| ファイル | 内容 | Git 管理 |
|---|---|---|
| `DOMAIN-CONTEXT.md` | 業界知識・一般的な慣習・汎用的な用語 | 管理対象 |
| `DOMAIN-CONTEXT.secret.md` | 特定取引先名・個別契約金額・社内ID・接続先URL等 | **`.gitignore` 対象** |

### .gitignore 設定

```gitignore
DOMAIN-CONTEXT.secret.md
**/DOMAIN-CONTEXT.secret.md
```

### 機密判定の目安

- 実在する企業名・個人名が含まれる → `.secret.md`
- 契約金額・料率・個別料金が含まれる → `.secret.md`
- 接続先の URL / ID / 認証情報が含まれる → `.secret.md`
- 「汎用的な業界知識」レベル → `DOMAIN-CONTEXT.md`

---

## INDEX.md からの参照

DOMAIN-CONTEXT.md が存在する場合のみ INDEX.md に以下を追加:

```markdown
## ドメイン文脈
→ DOMAIN-CONTEXT.md
（機密情報は DOMAIN-CONTEXT.secret.md に分離。Git 管理外）
```

非該当プロジェクトでは参照自体を省略（セクションを書かない）。

---

## 業界標準パックの扱い（射程外）

「finance-jp.md」「healthcare-jp.md」「retail-jp.md」等の**事前テンプレート**を本規格では**作成しない**。

### 理由
- フラクタル原則違反: 使われる前にテンプレートを量産するのは「AI 能力で動く」アプローチに反する
- 業界内でも業態差が大きく、汎用テンプレートは結局役立たない
- DOMAIN-CONTEXT.md は対話で埋まっていくべきもの

### 将来の扱い
- 複数プロジェクトで同業界の対話結果が蓄積された時、そこから**帰納的に**パターンが抽出される可能性はある
- その時点で検討する。現時点では実装しない

---

## L1 autonomous-dev との連携

- 存在する場合は L1 が INDEX.md 経由で DOMAIN-CONTEXT.md を読み込む
- `.secret.md` は L1 のコンテキストに注入する／しないはプロジェクト依存（機密ポリシー次第）
- 用語集（Cat-5）は L1 のコード・コメント・変数命名に影響する。**業界用語をコードに直訳しない**よう配慮

---

## 自己評価

体制事後評価で以下を蓄積:

- DOMAIN-CONTEXT.md 未作成で進めたが、実装時に業務理解不足で困ったか
- 5 カテゴリのどれが最も実装品質に寄与したか
- 機密分離ルールが守られたか

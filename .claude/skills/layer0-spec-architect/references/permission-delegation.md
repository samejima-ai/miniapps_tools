# 段階的権限委譲プロトコル

AIに委譲する権限の段階（level）と、AIから人間への介入・献上チャネルを規定する。
業務システム運用で「AI が勝手に何をやるか」「人間はどこで判断を差し込むか」を明示化するために導入。

---

## 原則

- 権限レベルは**段階的**であり、プロジェクト特性に応じて REGIME.md で明示する
- デフォルトは **L0-2**（最小権限）。REGIME.md に記載がなければ L0-2 と解釈
- 昇格は Layer 0 対話で明示同意を得た場合のみ。自動昇格は認めない
- L0-4（AI の暴走防止ガードを外す権限）は**不採用**。自己矛盾のため射程外
- 3 つの介入チャネル（C1/C2/C3）を通じて人間と AI が相互通信する

---

## 権限レベル定義

### L0-2（デフォルト）

| 項目 | 扱い |
|---|---|
| ローカルファイル編集 | 可 |
| テスト・ビルド実行 | 可 |
| git commit | **要人間承認** |
| git push | **不可**（人間が手動） |
| 本番デプロイ | 不可 |
| 外部 API 呼び出し（無料・read-only） | 可 |
| 外部 API 呼び出し（課金・write） | **要人間承認** |
| 破壊的操作（rm -rf / DB drop / force push） | 不可 |

**対応モード**: M1 / M2 のデフォルト。L2 でも配下 L1 は L0-2。

### L0-3（昇格）

| 項目 | 扱い |
|---|---|
| git commit | 可（CLAUDE.md / CHANGELOG 記述ルール順守前提） |
| git push | 可（保護ブランチ以外） |
| ステージング環境デプロイ | 可 |
| 本番デプロイ | **要人間承認** |
| 外部 API 課金・write | 可（上限額を REGIME.md に記録） |
| 破壊的操作 | 不可（引き続き） |

**昇格条件**:
- Layer 0 対話で人間が明示同意
- 可用性 A = 3 のプロジェクトでは REGIME.md に権限レベル記録が**必須**
- R = 3 のプロジェクトでは L0-3 への昇格を原則認めない（本番事故リスク回避）

### L0-4（**不採用**）

「破壊的操作・本番デプロイ・保護ブランチ書き換えを AI 単独で実行する」権限レベル。

- 本規格では**採用しない**
- 理由: AI の暴走検知が不可能な権限範囲であり、人間の判断献上チャネルが機能しなくなる（C1 が死ぬ）
- 将来 AI 能力が上がっても、本規格の射程外とする

---

## 3 つの介入チャネル

AI から人間への通信経路を 3 系統で明示する。

### C1: 判断献上チャネル（詳細規格）

AI が自走中に人間の判断を仰ぐための正式経路。**本規格で詳細を定める**。

#### 発動トリガー（5 カテゴリ）

| カテゴリ | 発動条件 | 例 |
|---|---|---|
| **Cat-A スコープ境界** | DONT.md に抵触する可能性があるとき | 新レイヤー追加、既存機能大幅改変 |
| **Cat-B 権限超過** | 現権限レベルで不可な操作が要件に含まれるとき | L0-2 で本番 DB migration 要求 |
| **Cat-C 外部依存** | 課金・契約・認証キー発行等、人間固有の責務 | API キー取得、有料 SaaS 契約判断 |
| **Cat-D 倫理／コンプライアンス** | 個人情報処理・法令関連の判断 | PII ログ出力の可否、同意取得 UX |
| **Cat-E 破壊的変更** | 既存データ・スキーマ・履歴を不可逆に書き換えるとき | DB カラム削除、INTENT.md 廃止 |

#### 献上フォーマット

```markdown
## 判断献上（C1）

- カテゴリ: [Cat-A 〜 Cat-E]
- 現状: [何が起きているか]
- 選択肢:
  1. [案 A] — メリット/リスク
  2. [案 B] — メリット/リスク
- AI 推奨: [どちらか。根拠を1行]
- 続行条件: [人間が返答しない場合の扱い。例: 30 分経過で Cat-A はブロック継続、Cat-C はデフォルト案採用]
```

REGIME.md の体制事後評価に判断献上履歴を記録する。

### C2: 割り込みチャネル（存在規格のみ）

**人間 → AI の能動的介入**。AI 実行中に人間が割り込む。

- 規格上の存在のみ定義。具体実装はプロジェクト依存（IDE 機能・dialog-harness-layers 機能）
- REGIME.md の体制事後評価に割り込み履歴を記録する
- 具体実装は**射程外**

### C3: 通知チャネル（存在規格のみ）

**AI → 人間の能動的通知**。判断献上ではない情報共有（進捗・警告・異常）。

- 規格上の存在のみ定義。具体実装はプロジェクト依存
- 通知内容の定型化・通知先（Slack / メール等）・頻度制御は規定しない
- 具体実装は**射程外**

---

## REGIME.md への記録フォーマット

```markdown
## 権限委譲設定

- 権限レベル: [L0-2 / L0-3]（未記載時は L0-2）
- 昇格経緯: [デフォルト / Layer 0 対話で人間明示同意（日付）]
- 外部 API 課金上限: [額 / 無制限 / 非該当]
- 介入チャネル:
  - C1（判断献上）: 有効（本規格準拠）
  - C2（割り込み）: [プロジェクト側実装 / 未実装]
  - C3（通知）: [プロジェクト側実装 / 未実装]
```

## オーバーライドルールとの連携

`nfr-scoring.md` のオーバーライド条件と整合:

- **A (Availability) = 3**: 権限レベルの明示記録を必須化（未記載は REGIME.md 不完全扱い）
- **R ≥ 2 / S ≥ 2**: L0-3 昇格は特に慎重な判断を要求（強制禁止ではない）
- **N ≥ 9**: ARC 選択に加えて権限レベルの再評価を Layer 0 に指示

---

## 運用インシデントと L3 運用層の関係

本プロトコルは **L3 運用層を新設しない**。
運用インシデントが発生した場合の扱い:

- インシデント発生 → **新仕様発見**として扱う
- L0 対話に戻り SPEC.md / REGIME.md を更新
- L1 再起動で対応実装

「運用だけ L3 として独立させる」設計は採らない（フラクタル原則違反のため射程外）。

---

## 自己評価

体制事後評価で以下を蓄積:

- L0-2 のままで十分だったか / 昇格が正解だったか
- C1 判断献上の頻度（多すぎ = 仕様不足、少なすぎ = スコープ逸脱の見落とし）
- C2 割り込みが発生した場面（仕様曖昧さの兆候）
- 各 Cat-A 〜 Cat-E カテゴリの発動妥当性

# サブフェーズ共通対話プロトコル

L0 の各サブフェーズ（L0-2〜L0-6）が従う共通の対話・生成・検証・判定プロトコル。
本ファイルは骨格のみを定義し、サブフェーズ固有の対話カテゴリ・生成物フォーマットは各 `subphase-l0X-*.md` に委ねる。

---

## 原則

- 各サブフェーズは「対話 α → 生成 β → 検証 γ → 判定 δ」の 4 フェーズで構成する（ミニ L0→L1→L2 構造）
- 人間は「語る」と「決める」のみ。AI が生成・検証・提示を担う
- 各サブフェーズは独立した AI 呼び出し単位とする（トークン効率と局所リトライのため）
- 前サブフェーズの成果物を**入力**として読み込み、型名・状態名・エンティティ名を後続へ伝搬する
- 1 サブフェーズの対話で 5 問を超えたら分割する（既存 `dialog-questions.md` の「詰め込み禁止」原則を継承）

---

## 儀式との違い

`ritual-protocol.md` と本プロトコルは目的と軸が異なる。混同しないこと。

| 観点 | 振り返り儀式 | サブフェーズ共通プロトコル |
|---|---|---|
| 軸 | 時系列整合（過去文脈⇔現欲求） | 構造的整合（層間の入出力契約） |
| 発動タイミング | ステップ 1.5（LC ≥ 1） | ステップ 3.5（該当プロジェクトのみ） |
| 担当範囲 | history/ の読込と矛盾検出 | spec/ の生成と不整合検出 |
| 人間役割 | 気付き役（AI が検出・提示・記録） | 決定役（AI が質問・生成・検証） |

## 4 フェーズ詳細

### Phase α: 対話（AI 質問 → 人間回答）

- AI は当該サブフェーズの「対話カテゴリ」（各 `subphase-l0X-*.md` に定義）に沿って質問を投げる
- 非エンジニア語彙を用いる（既存 `dialog-questions.md` の原則を継承）
- 1 回の応答で質問を詰め込まない（2〜3 問まで）
- 曖昧回答は **E1 充当**（`ritual-protocol.md` 準拠）で「AI 推定 + 通知型」で埋め、`subphase-manifest.md` に `確度: AI推定` を付与
- 曖昧回答 2 回連続は「重要度低」と解釈し、簡易モードに自動格下げ

### Phase β: 生成（AI が成果物ファイル作成）

- 成果物は `spec/` ディレクトリ配下に配置（各サブフェーズで固定ファイル名、後述「成果物配置規約」参照）
- 冒頭コメントに必ず以下を記載:
  - サブフェーズ ID（例: `// L0-2: Domain Model`）
  - モード（完全 / 簡易）
  - 生成日（YYYY-MM-DD）
- 前サブフェーズの成果物を**改変しない**。ズレを検出したら Phase γ で該当サブフェーズへ差し戻す

### Phase γ: 検証（AI が前サブフェーズとの不整合を検出）

- 各サブフェーズの「検証フェーズ」節（各 `subphase-l0X-*.md`）に記載の観点でクロスチェック
- 共通観点:
  - 型名・enum 値が前サブフェーズと一致
  - SPEC.md の登場名詞が抜け落ちていない
  - 命名規約（キャメルケース / ケバブケース等）が一貫
- 不整合を検出したら:
  - 軽微（命名揺れ等）→ AI が自動修正し、`subphase-manifest.md` に修正ログ
  - 重大（仕様矛盾）→ 該当サブフェーズに差し戻す。人間には矛盾箇所のみ提示

#### Phase γ-i: 業界叡智照合フック（W4-Q2 採決追加、CTL ≥ 1）

Wave 4 W4-Q2 採決 (`council-2026-05-11T19:30:00Z-w4qb02`、段階組込 B、conf 0.78) で確定した業界叡智参照モード組込機構。Phase γ 本流（共通観点クロスチェック）の **補強層** として動作し、L0 対話で生成された SPEC ドラフトが業界叡智 (ECC / hooks / AgentShield 等) と整合的かを観測駆動で照合する。

**第 8 条 3 段階モデル準拠**（W4-Q1 採決で再確認、minority C は Wave 5 温存）:

- **観測**: SPEC ドラフトと業界叡智 (`history/refs-draft/ecc/` 配下 5 ファイル / `templates/rules/common/agentshield-reference.md` / `.claude/hooks.json` schema 等) の照合
- **候補化**: 不整合・補強候補をリスト化、L0 対話の判断材料として提示（自動採用なし）
- **人間最終承認**: Phase δ でユーザー承認 (philosophy 第 6 条「人間最終承認」)

**起動条件 (CTL 連動)**:

| CTL | 動作 |
|---|---|
| CTL 0 | inactive（観察温存、候補化も抑止） |
| CTL ≥ 1 | active、候補出力のみ（採用判断は人間） |

**出力フォーマット**（候補リストのみ、自動採用なし）:

```yaml
industry_wisdom_match_candidates:
  - source: "ECC agents-pattern.md"  # 参照ソース
    aspect: "agents 定義の YAML frontmatter (tools/model 指定)"
    spec_draft_reference: "SPEC.md §3.2 agent 定義節"
    match_type: "complementary"  # complementary / contradictory / redundant
    suggestion: "tools 指定で読み取り専用 agent を明示する慣例あり、検討候補"
    confidence: 0.7
```

**運用観測項目（Wave 4 末振り返り儀式で集計）**:

- W4-Q1 採決で必須化された 3 観測項目（Council 経由率 / 3 段階運用実績 / minority C 再評価データ蓄積）
- 「業界叡智参照を経た SPEC の unedited merged 率」（W4-Q2 哲学者 concerns 由来、DH 哲学独占性希釈リスク監視）
- 「業界叡智照合フック起動回数 / 候補リスト採用率 / 却下率」

**Wave 5 申し送り**（W4-Q2 stance A フル組込の繰越分）:

- `subphase-l03-api.md` / `l04-transition.md` / `l05-authz.md` / `l06-invariants.md` / `scaffold-checklist.md` の個別改修
- 各 subphase の業界叡智参照モード詳細仕様

**関連諮問**:

- `council-2026-05-11T19:00:00Z-w4qb01`（W4-Q1、3 段階モデル維持確定）
- `council-2026-05-11T19:30:00Z-w4qb02`（W4-Q2、段階組込 B 確定、全会一致 conf 0.78）

### Phase δ: 判定（人間レビュー・承認）

- AI は Phase β/γ の結果を**差分サマリ**で提示（全文は提示しない）
- 提示フォーマット:
  - 生成ファイルとモード（完全 / 簡易）
  - 前サブフェーズからの継承要素（型名・状態名等）
  - γ で検出・修正した不整合（あれば）
  - **γ-i 業界叡智照合候補**（CTL ≥ 1 のみ、`industry_wisdom_match_candidates` リスト。0 件なら省略可。採用判断は人間最終承認、自動採用なし）
  - 次サブフェーズへの引き渡し要素
- 人間の応答:
  - 承認 → 次サブフェーズへ
  - 修正要求 → Phase α へ戻る（該当サブフェーズ内ループ）
  - 前サブフェーズ再実行 → 該当サブフェーズから再開

## 成果物配置規約

```
project-root/
├── SPEC.md              # L0-1 の成果物（既存）
└── spec/                # サブフェーズ起動時のみ作成
    ├── subphase-manifest.md  # 選定結果・確度・起動ログ
    ├── domain.ts             # L0-2 起動時
    ├── api.tsp               # L0-3 完全モード時
    ├── api-signatures.ts     # L0-3 簡易モード時
    ├── state-machine.ts      # L0-4 完全モード時
    ├── state-diagrams.md     # L0-4（完全/簡易問わず）
    ├── authz.fga             # L0-5 完全モード時
    ├── authz-matrix.md       # L0-5 簡易モード時
    └── invariants.feature    # L0-6 起動時
```

**原則**: `spec/` はサブフェーズ起動時のみ作成する。全サブフェーズがスキップのプロジェクトでは `spec/` 自体を生成しない。

## サブフェーズ間の入出力契約

各サブフェーズが後続に渡す要素を一覧化。各 `subphase-l0X-*.md` の「前サブフェーズからの入力」「後続サブフェーズへの出力」節はこの表と整合すること。

| 生成元 | 継承される要素 | 継承先 |
|---|---|---|
| L0-1 `SPEC.md` | WHAT 節の名詞（エンティティ候補）・条件（上限値・バリデーション）・制約 | L0-2（全て）、L0-3〜L0-6（参照） |
| L0-2 `domain.ts` | 型名・enum 値・集約上限値・値オブジェクト | L0-3（Model）、L0-4（状態名）、L0-5（type）、L0-6（Given 節） |
| L0-3 `api.tsp` | エンドポイント一覧・エラーコード・HTTP ステータス | L0-5（認可対象）、L0-6（When 節） |
| L0-4 `state-machine.ts` | 状態名・遷移イベント・初期状態 | L0-6（状態遷移 Feature） |
| L0-5 `authz.fga` | 関係名（owner/can_read 等）・type 定義 | L0-6（Evil Path Scenario） |

## スキップ条件と簡易モード

各サブフェーズは起動判定時に以下の 3 状態のいずれかを取る（判定ロジックは `subphase-selection.md` 参照）。

| 状態 | 意味 | 後続への影響 |
|---|---|---|
| **完全モード** | フル成果物生成、OSS ツール活用 | 後続に型名・状態名等を完全に引き渡す |
| **簡易モード** | 軽量成果物（マトリクス・関数シグネチャ・Mermaid のみ等） | 後続は簡易モード情報で代替。参照は可能 |
| **スキップ** | 当該サブフェーズを完全に実行しない | 後続での参照を**無効化**（例: L0-5 スキップ時は L0-6 で Evil Path を書かない） |

**スキップ時の後続ルール**:
- L0-2 スキップ → L0-3 は完全モード起動不可（簡易モードのみ）
- L0-5 スキップ → L0-6 は Evil Path Scenario を書かない
- L0-6 はそれ単独では起動しない（2 以上のサブフェーズ起動時のみ発動）

## 事後追加時の再実行プロトコル

プロジェクト進行中にサブフェーズの追加・昇格・訂正が必要になった場合のフロー。詳細は `subphase-selection.md` の「事後追加プロトコル」を参照。

本プロトコルでは共通ルールのみ定義:

- 事後追加されたサブフェーズは**独立した AI 呼び出し**で実行する
- 依存関係チェック（L0-5 追加なら L0-2/L0-3 を参照できること）を先に行う
- 事後追加・昇格・訂正の全てについて `subphase-manifest.md` に `事後変更ログ` を追記する
- L0-6（不変条件）は後続全サブフェーズの変更を受けるため、事後変更時は**必ず再生成**する

## L1 との連携

L1 autonomous-dev は `spec/` 配下の成果物を以下のように利用する（Phase 3 で正式化予定、Phase 1 では参考情報）。

- `domain.ts` → そのまま実装コードで import
- `api.tsp` → OpenAPI 生成後、コントローラ雛形に利用
- `state-machine.ts` → XState ランタイムで利用可
- `authz.fga` → OpenFGA サーバーへの投入
- `invariants.feature` → L1 独立検証のチェックリストとして参照

L1 独立検証（`layer1-independent-reviewer`）は `invariants.feature` の Scenario を網羅的に検証する（Phase 3 改訂で正式化）。

## プロトコル自己評価

本プロトコルは以下のシグナルで再設計の対象となる:

- 4 フェーズのいずれかが形骸化（例: Phase γ で検出が毎回ゼロ）→ 検証観点の不足を疑う
- サブフェーズ間の I/O 契約が複数プロジェクトで破綻 → 契約表の見直し
- 事後追加が頻発（誤判定率 > 30%）→ `subphase-selection.md` の判定表見直し
- 人間の「承認疲れ」（Phase δ で判定なく承認されるケース多発）→ 差分サマリ提示方法の改善

評価は Phase 5（REGIME-LOG 連携）で REGIME-LOG.md に蓄積し、次サイクルの判定精度向上に利用する。

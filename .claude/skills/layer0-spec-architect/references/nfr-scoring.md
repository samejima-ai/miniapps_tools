# NFR スコア規格

非機能要件（Non-Functional Requirements）を S/U/R と並ぶ第4軸 **N** として定量化するためのスコアリングプロトコル。
業務システム運用・本番サービス型プロジェクトで、体制判定が「何となく M2」に収束する問題を避けるため導入する。

`regime-assessment.md` の S/U/R と同じ判定フローに組み込まれる。

---

## 原則

- NFR は 5 カテゴリ × 0〜3 点で定量化し、合計 0〜15 点を N スコアとする
- 各カテゴリは**存在するかどうか**でまず 0 / 1 点以上を分ける。**強度**に応じて 2〜3 点へ積む
- 曖昧応答は **0 点で開始**する（`REGIME.md` の NFR スコアセクションに記録がない場合のデフォルト = N=0 と同値）
- 4 つのオーバーライドルールで体制判定を上書きする（下記）
- 業界固有のコンプライアンス要件は DOMAIN-CONTEXT.md 側で拾い、N スコアでは「存在する／しない」の粒度に留める

---

## 5 カテゴリの配点

### P: 性能（Performance）

| スコア | 条件 |
|---|---|
| 0 | レイテンシ・スループット要件なし |
| 1 | 体感品質（秒単位の応答）のみ明示 |
| 2 | レイテンシ SLO（p95 〜 ms/s）が数値指定されている |
| 3 | 高スループット（大量同時接続・大容量データ処理）＋レイテンシ両立が要件 |

### A: 可用性（Availability）

| スコア | 条件 |
|---|---|
| 0 | 落ちても困らない（実験・自分用） |
| 1 | 平日日中のみの内部ツール水準 |
| 2 | 99.9% 相当、業務時間内停止が痛い |
| 3 | 99.95% 以上 or 24/7 運用、停止が金銭・安全影響 |

### S: セキュリティ（Security）

| スコア | 条件 |
|---|---|
| 0 | 認証不要・公開情報のみ |
| 1 | ログイン必須、内部情報を扱う |
| 2 | PII / 個人情報 / 決済情報 / 医療情報のいずれかを扱う |
| 3 | 規制業界（金融・医療・公共）＋外部攻撃面あり |

### C: コンプライアンス（Compliance）

| スコア | 条件 |
|---|---|
| 0 | 特別な規制なし |
| 1 | 社内規定・業界ガイドライン程度 |
| 2 | 法令準拠（個人情報保護法・業法）が明示要件 |
| 3 | 複数規制の同時準拠（例: 個人情報保護法 + 金商法 + 業界固有）or 監査必須 |

### L: スケーラビリティ（Scalability）

| スコア | 条件 |
|---|---|
| 0 | 単一環境・固定規模で足りる |
| 1 | 将来の拡張を意識するが現時点は単一構成 |
| 2 | 水平スケール前提、負荷変動吸収が要件 |
| 3 | マルチリージョン・グローバル分散・バースト対応が要件 |

---

## N スコアの合計と意味

合計値の範囲: **N = 0 〜 15**

| N 範囲 | 典型像 |
|---|---|
| 0〜2 | 非機能要件ほぼなし。実験・私用・社内ツール初期 |
| 3〜4 | 内部運用ツール、停止や漏洩が業務影響あるレベル |
| 5〜8 | 業務システム運用・本番サービス。運用設計が必要 |
| 9〜15 | ミッションクリティカル。アーキテクチャ選定段階から NFR 駆動 |

N は**判定を縛るのではなく、上書きルールの発動条件**として使う。単独で M2→L2 等の強制はしない。

---

## オーバーライドルール（4 条件）

N 軸に由来する体制判定の強制上書き。`regime-assessment.md` のオーバーライド優先順位表に組み込まれる。

### ルール1: N ≥ 9 → モノリス解除を強制

- ARC デフォルト `monolith` を **強制解除**し、`arc-patterns/` 配下から人間に選択させる
- N ≥ 9 はミッションクリティカル帯。モノリス単独では可用性・スケーラビリティの両立が難しいため
- 選択は人間の判断献上で行う（AI は推奨のみ）

### ルール2: N ≥ 5 → 推奨切替検討（2段階化）

- **ARC 選択を人間に提案**する（強制ではない）
- S/U/R 一次判定が M1 でも **M2 以上を推奨**する（強制は既存 R≥2 ルール側）
- 運用設計の必要性をドキュメントで指摘する（DONT.md に L3 運用層を先に置かないことを明記）

### ルール3: セキュリティ ≥ 2 → M2 強制

- S (security) カテゴリが 2 点以上なら **M2 以上を強制**（M1 不可）
- 独立検証（layer1-independent-reviewer）を必須化するため
- 既存の R ≥ 2 → M2 強制ルールと**競合しない**（両方発火しても結果は M2 で一致）
- ただし、R=1 かつ S=2 の組合せで本ルールが単独発火する（金銭関与なしでも PII ありなら M2 強制）

### ルール4: 可用性 = 3 → 権限レベル記録必須

- A (availability) カテゴリが 3 点（24/7・ミッションクリティカル）なら REGIME.md に**権限レベルの明示記録を必須**化する
- デフォルト `L0-2`（`permission-delegation.md` 参照）。明示しないと REGIME.md が不完全と判定される
- `permission-delegation.md` の昇格条件と齟齬なし

## オーバーライド優先順位（N 軸を組み込んだ完全版）

`regime-assessment.md` 既存の優先順位表に N 系を追記した統合版:

1. **L2 発動閾値** → L2 確定（最優先）
2. **U ≥ 3** → L0 対話延長（モード確定保留）
3. **N ≥ 9** → モノリス解除強制（ARC 選択を人間判断献上）
4. **N ≥ 5** → ARC 推奨切替 + 2段階化推奨（強制ではない）
5. **R ≥ 2** → M2 以上強制
6. **セキュリティ ≥ 2** → M2 以上強制
7. **可用性 = 3** → 権限レベル記録必須化

ルール 5・6 は同じ結果（M2 強制）に収束するため衝突しない。ルール 3 は ARC を上書きするが M2/L2 判定自体は上書きしない。

## 判定フロー（N 軸追加後）

```
1. 対話から S, U, R, N を算出
2. L2発動閾値を超える？
   YES → L2 確定
3. U ≥ 3 ?
   YES → L0 対話延長
4. N ≥ 9 ?
   YES → ARC モノリス解除。arc-patterns/ から人間選択を献上
5. N ≥ 5 ?
   YES → ARC 選択を推奨、2段階化を検討
6. S + U + R で一次判定（0〜3=M1 / 4〜=M2）
7. R ≥ 2 ? または N の security ≥ 2 ?
   YES → M2 に引き上げ
8. N の availability = 3 ?
   YES → 権限レベル明示記録を REGIME.md に必須化
9. 確定 → REGIME.md 出力
```

## デフォルトと後方互換

- **`REGIME.md` に NFR スコアセクションがないプロジェクト**は **N=0** として扱う（全オーバーライド非適用）
- 既存 REGIME.md 形式（S/U/R のみ）のプロジェクトはそのまま通る
- 本規格は**新規プロジェクトと、機能追加時に NFR 情報が引き出せた既存プロジェクトのみに適用**する
- Layer 0 対話で N 軸の情報が引き出せなかった場合は N=0 を明示記録し、次サイクル以降で精緻化する

## ガバナンス補助観点（命名規則・インデックス・パーティション）（v5.17.0 追加）

NFR の P (Performance) / L (Scalability) 軸の判定材料として、データ層のガバナンス・最適化観点を補助情報として明文化する。これらは独立カテゴリではなく、P / L のスコア根拠を引き出す補助観点として L0 対話で取得する。

### 命名規則とデータディクショナリ

**判定: P サブスコアの補強根拠**

人間 + AI 両者が解釈しやすいスキーマを保つガバナンス。AI 自律開発が前提の DH では特に重要：

- **命名一貫性**: snake_case（DB 列）/ camelCase（TS）/ kebab-case（URL）の用途別徹底
- **識別子の明瞭性**: `data` / `info` / `value` 等の意味の薄い名前を避ける（`user_id` / `order_total_jpy` のように単位・対象を含める）
- **データディクショナリ**: 全列の意味・単位・制約を `SPEC.md` または `spec/domain.ts` に整備（AI が後続セッションで参照可能な構造化形式）

L0 対話シグナル：
- 「同じ概念を複数の名前で呼んでいる」発話 → 命名規則整備の必要性
- 既存プロジェクト（LC ≥ 1）で archeo-architect 起動推奨

### インデックス戦略

**判定: P サブスコア判定の supporting evidence（補強根拠）**

P スコア表（上記）は「レイテンシ SLO の数値指定」を P=2 の必要条件とする。インデックス戦略の必要性は P スコア表を**上書きしない**が、SLO 達成手段としての具体策の有無を確認する補強情報として扱う。クエリパターンに合わせたインデックス設計：

| パターン | インデックス種別 |
|---|---|
| 等価検索（`WHERE col = ?`）| B-tree（デフォルト） |
| 範囲検索（`WHERE col > ? AND col < ?`）| B-tree |
| 複数列の組み合わせ検索 | 複合インデックス（順序に注意、最左前置） |
| 全文検索 | GIN / tsvector |
| JSON 内検索 | GIN（JSONB） |
| 部分検索（`WHERE col LIKE 'prefix%'`）| B-tree（前方一致のみ可、後方一致は GIN trigram） |

**トレードオフ**: 過剰インデックスは書き込み性能劣化と容量肥大を招く。読み書き比率と頻度をクエリ別に把握して必要分のみ作成する。L1 実装で `EXPLAIN ANALYZE` を sensors/computational.md に組み込み、未使用インデックス検出を運用に組み込む。

### パーティショニング戦略

**判定: L サブスコア判定の supporting evidence（補強根拠）**

L スコア表（上記）は「水平スケール前提 / 負荷変動吸収が要件」を L=2 の必要条件とする。データ量増大予測やパーティション必要性は L スコア表を**上書きしない**（水平スケール要件と必ずしも等価ではない）。L=1 段階の「将来拡張意識」根拠としての補強情報、または L=2 確定時の実装手段提示として扱う。3 つの軸：

| 軸 | 適用場面 |
|---|---|
| **時系列パーティション**（月別 / 年別） | ログ・イベント・履歴データ、古いデータの段階的削除 |
| **テナント別パーティション** | マルチテナンシー + テナント数が安定（→ `subphase-l05-authz.md` §マルチテナンシー隔離戦略 と連動） |
| **レンジパーティション**（ID 範囲 / ハッシュ） | 単一巨大テーブルの分割、シャーディングの前段 |

パーティショニングは事後追加が困難（既存データの再分配が必要）。L0 で「データ量が将来どこまで増えるか」の見立てを取得し、L サブスコア ≥ 2 と判定されたら設計段階で組み込む。

### シャーディング（水平分散）

**判定: N ≥ 9 のオーバーライド対応**

データを複数 DB インスタンスに水平分散する。N ≥ 9（モノリス解除強制）のシグナルとして観測される：

- シャードキー選定が後戻り困難（user_id / tenant_id / region_id 等）
- クロスシャードクエリは原則禁止または性能犠牲
- DB 分離型マルチテナンシー（`subphase-l05-authz.md`）と類縁

シャーディング採用は ARC = realtime-pubsub / event-sourcing と組み合わせることが多い。`arc-patterns/` の人間判断献上で決定する。

### 対話シグナルへの組み込み

以下は本ファイル（`nfr-scoring.md`）内で提示する追加対話シグナル例。`dialog-questions.md` を機械的に書き換える対象ではなく、AI が NFR 対話を深掘りする際の参考として参照する（必須ではない）：

| 質問 | 引き出す情報 |
|---|---|
| 「データの量はどれくらい増える見込みですか？」 | パーティション / シャーディング判定材料 |
| 「よく検索する条件・並び順は何ですか？」 | インデックス戦略の判定材料 |
| 「同じ概念が複数の名前で呼ばれていることはありますか？」 | 命名規則整備の必要性、archeo-architect 起動シグナル |

非エンジニア対話では技術用語（インデックス / パーティション）を出さず、「速くしたい操作」「将来どこまで増えるか」「同じ意味で違う言葉が混ざっていないか」の日常語で聞く。

### REGIME.md への記録（任意セクション）

ガバナンス観点は P / L サブスコアの根拠として記録する。独立カテゴリ化は破壊的変更になるため見送り：

```markdown
## NFRスコア（N）

- 性能 P: 2  # 根拠: 応答時間 SLO 数値指定あり（p95 < 300ms）。インデックス戦略は SLO 達成手段の supporting evidence
- 可用性 A: ...
- ...
- スケーラビリティ L: 1  # 根拠: 将来拡張意識、データ量増大予測あり。水平スケール要件未確定のため L=1（時系列パーティション設計は L=1 段階の supporting evidence）

## ガバナンス補助観点（任意）

- 命名規則: snake_case 統一、データディクショナリ整備済み
- インデックス戦略: 頻繁検索パターン 3 種特定、複合インデックス検討要
- パーティション戦略: 時系列（月別）を L サブスコア根拠として採用
```

## プロトコル自己評価

体制事後評価（`delivery-format.md`）で N 軸の妥当性も蓄積する:

- 過剰：N=6 で M2 強制したが独立検証が無指摘だった → N の重みを見直し
- 過少：N=3 で M2 通過したが運用で事故った → 判定表にカテゴリ追加
- オーバーライド妥当性：N ≥ 9 のモノリス解除が実際に必要だったか
- ガバナンス観点：命名規則・インデックス・パーティション戦略が P / L サブスコアの根拠として正しく対話で引き出せていたか

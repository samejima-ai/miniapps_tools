# 対話質問例集

Layer 0（spec-architect）が非エンジニアから体制判定情報を引き出すための質問例。
技術用語を使わず、人間が直感的に答えられる形で設計する。

---

## 原則

- 技術用語を避ける（「SLA」「認証」「永続化」等は使わない）
- 具体的な数字や選択肢を提示する
- 1回のターンで質問を詰め込まない（最大2〜3問）
- 「分からない」と答えられる余地を残す
- 仕様の具体化と自然に織り交ぜる

---

## 規模スコア（S）を引き出す質問

### 機能数の把握

避ける：「機能はいくつですか？」

推奨：
- 「このツールで『できること』を思いつく限り挙げるとどんな感じ？」
- 「『〜できたらいいな』と思うことを箇条書きにするとどれくらいになりそう？」
- 「1〜3個くらい？ 4〜10個くらい？ それ以上？」

### 画面数の把握

避ける：「画面構成は？」「UIは？」

推奨：
- 「使う人が見る画面って、どんな画面がありそう？」
- 「設定画面とメイン画面、みたいに分かれてる？」
- 「画面が必要なやつ？ それとも裏で動くだけ？」

### 登場人物の把握

避ける：「ユーザーロールは？」「権限は？」

推奨：
- 「使う人はどんな人？ 自分だけ？ 他の人も使う？」
- 「管理する人と使う人って分かれてる？」
- 「全員同じことができればいい？ それとも人によって見える範囲が違う？」

### 外部連携の把握

避ける：「API統合は？」「外部サービスは？」

推奨：
- 「他のサービスとつながる必要ある？ 例えばGmailとかSlackとか」
- 「データをどこかから取ってくる？ どこかに送る？」
- 「全部このツールの中で完結する？」

---

## 不確実性スコア（U）を引き出す質問

### イメージの明確さ

避ける：「仕様は固まってますか？」

推奨：
- 「頭の中でどれくらい絵が見えてる？ 具体的？ ぼんやり？」
- 「完成したときの画面、思い浮かべられる？」
- 「『こういう感じ』って他の人に説明できる？ それともまだ探り探り？」

### 既知度

避ける：「技術選定は？」「類似システムは？」

推奨：
- 「似たようなもの、使ったことある？」
- 「『こういうの欲しい』って思ったきっかけは、何か似たサービス？」
- 「全く初めての領域？ それとも普段触ってるジャンル？」

### 変更予測

避ける：「ロードマップは？」「保守計画は？」

推奨：
- 「1回作って終わり？ それとも使いながら直していく感じ？」
- 「完成したら放置？ それとも機能追加していく予定？」

---

## リスクスコア（R）を引き出す質問

避ける：「SLAは？」「可用性要件は？」「セキュリティ要件は？」

推奨：
- 「壊れたらどれくらい困る？」
- 「使うのは自分だけ？ 他の人にも渡す？」
- 「お金が絡む？ それとも趣味・実験の範囲？」
- 「動かなくなっても笑える？ それとも困る？」

## 対話パターン例

### パターン1：最初の一言で情報が薄いケース

ユーザー：「タブマネージャーみたいなChrome拡張を作りたい」

対話の流れ：
1. まず目的と全体像を確認
   「どんな悩みを解決したい感じ？」
2. 機能の広がりを掴む
   「『できること』を思いつく限り挙げてみて」
3. リスク確認
   「これ、自分だけ使う？ Chrome Web Storeに公開する感じ？」
4. 画面の有無
   「設定画面みたいなのはいる？」

このパターンで S=3, U=1, R=2 程度の情報が集まる。
合計 6点 + R≥2 → **M2 標準モード**（L1 + layer1-independent-reviewer）で運用。

### パターン2：「なんとなく欲しい」だけのケース

ユーザー：「SaaSっぽいの作りたい」

対話の流れ：
1. まず具体化を試みる
   「どんな悩みを解決したい？ 誰の悩み？」
2. 既知度を探る
   「似たサービス使ってて『こうだったらいいのに』って思った？」
3. 答えが出ないなら不確実性を認める
   「じゃあまずイメージを一緒に固めるところから始めよう」

このパターンで U≥3 を検出した時点で **L0 対話延長**を宣言。
モード確定は保留し、仕様が煮詰まったら S/U/R を再算出する。
早く形が欲しい場合は暫定 M1 で小さく試作し、使いながら仕様を固めるのも可。

### パターン3：鮮明だが規模が読めないケース

ユーザー：「社内のタスク管理ツール作りたい」

対話の流れ：
1. 画面構成を掴む
   「使う人が見る画面は？ 一覧とか詳細とか」
2. 登場人物を掴む
   「使う人は何種類？ 管理者と一般メンバー、みたいな？」
3. 外部連携を掴む
   「SlackとかGmailとつなぐ必要ある？」
4. リスク確認
   「社内で本番運用する感じ？ 試作？」

このパターンで S=5〜7, R=2〜3 の範囲が見えてくる。
- S=5〜6 かつ L2閾値未満 → **M2 標準モード**
- S=7以上 + 独立ドメイン数≥5（ユーザー管理/タスク/通知/レポート/権限 等）→ **L2 発動**を検討
- 独立ドメイン数が閾値未満なら M2 で吸収

## 質問の順序

体制判定情報を取得する順序は以下を推奨：

1. **目的・やりたいこと**（仕様具体化と共通）
2. **機能の広がり**（規模S）
3. **登場人物**（規模S）
4. **画面の有無**（規模S）
5. **既知度・類似物の経験**（不確実性U）
6. **イメージの明確さ**（不確実性U。4までの答え方から推測も可能）
7. **リスク**（リスクR。最後に確認）
8. **外部連携**（規模S。リスクと絡めて聞くと自然）
9. **変更予測**（不確実性U。最後でよい）

後半の項目は前半の回答から推測できる場合は質問を省略する。
判定プロトコルの完全性より対話の自然さを優先する。

## L0 サブフェーズ選定質問

ステップ 3.5 で起動する動的サブフェーズ（L0-2 ドメインモデル / L0-3 API 契約 / L0-4 状態遷移 / L0-5 認可 / L0-6 層間不変条件）を選定するための 5 問。
判定プロトコル全体は `subphase-selection.md` を参照。

本 5 問は**ステップ 2 の対話中に自然に織り交ぜる**。専用の質問タイムを設けず、モード判定情報の取得と並行して聞き取る。
全問が「不要」に倒れればステップ 3.5 は完全スキップされ、従来フロー（ステップ 1→7.6）と同一挙動となる。

### S1: 永続化（→ L0-2 ドメインモデル）

- 「データを保存する必要はありますか？ データベースを使う感じ？」
- 「それとも画面を閉じたら消えて大丈夫？」

**解釈**:
- DB 使用 or 複雑なデータ構造の永続化 → **L0-2 完全モード**
- メモリ上のみだが複雑なデータ構造 → **L0-2 簡易モード**
- ステートレス（CLI ワンショット等）→ **L0-2 スキップ**

### S2: 外部連携（→ L0-3 API 契約）

- 「外部のシステムや API とつなぐ必要はありますか？」
- 「つなぐ先は何ですか？（ブラウザから / 他のサーバーから / モバイルアプリから）」

**解釈**:
- 外部公開 or 複数クライアント → **L0-3 完全モード（TypeSpec）**
- 内部関数呼び出しのみ → **L0-3 簡易モード（関数シグネチャ）**
- API 境界なし（CLI 単体等）→ **L0-3 スキップ**

### S3: 画面と遷移（→ L0-4 状態遷移）

- 「画面はいくつくらいありますか？ 画面同士の移動は複雑？」
- 「ステータスが変わるもの（承認中 → 完了 みたいな）はありますか？」

**解釈**:
- 画面遷移が複雑 or エンティティのライフサイクル定義あり → **L0-4 完全モード（XState + Mermaid）**
- 画面少・業務状態はフラグのみだが可視化したい → **L0-4 簡易モード（Mermaid のみ）**
- 状態概念なし → **L0-4 スキップ**

### S4: ユーザーと権限（→ L0-5 認可）

- 「複数ユーザーで使いますか？ 権限の違いはありますか？」
- 「自分が作ったものしか見られない、とか決まりはありますか？」

**解釈**:
- 複数ユーザー + 所有権 or 共有機能 → **L0-5 完全モード（OpenFGA DSL）**
- 複数ユーザーだが権限は固定 → **L0-5 簡易モード（マトリクス）**
- シングルユーザー → **L0-5 スキップ**

### S5: 自動遷移・承認（→ L0-6 層間不変条件）

- 「時間経過や承認で状態が自動的に変わるものはありますか？」
- 「『この条件を破るとシステムが壊れる』という約束事はありますか？」

**解釈**:
- L0-2〜L0-5 のうち **2 つ以上が起動** → **L0-6 起動**（Happy/Sad/Evil 三分類）
- 起動サブフェーズが 0〜1 個 → **L0-6 不起動**（単独では意味を持たない）
- L0-5 がスキップ → L0-6 起動しても **Evil Path Scenario は書かない**

### 曖昧回答の扱い

回答が曖昧（「たぶん」「どうだったかな」「どっちでもいい」等）なら、`ritual-protocol.md` の E1 対応に準じて **AI 充当 + 通知型** で暫定値を置く。

- 暫定値を `spec/subphase-manifest.md` の `確度: AI推定` 欄に記録
- 「〇〇と推定しました。違ったら次回教えてください」と通知のみ（再質問しない）
- 曖昧回答が **2 回連続**発生したら、当該サブフェーズを自動的に**簡易モードに格下げ**（記憶が薄い領域＝重要度低の推定）

### 既存プロジェクトへの後方互換

ステップ 3.5 は新規起動の L0 対話のみで発動する。
`spec/subphase-manifest.md` が存在しない既存プロジェクトでは本 5 問をスキップし、従来フローを維持する。

## 振り返り儀式の質問テンプレ（LC ≥ 1 で使用）

履歴層 `history/` がある既存プロジェクトで、L0 対話の冒頭に実行する振り返り儀式のテンプレ。
儀式プロトコル全体は `ritual-protocol.md` を参照。

### F1: 過去文脈サマリ提示（質問は最大1問）

SUMMARY.md を提示した上で：

- 「前回ここまで進みました（要約）。今回はこれの続きで合ってますか？」

それ以上は聞かない。F1 の目的は認識共有のみ。

### F2: 認識ズレ検出（検出時のみ質問）

AI が過去 INTENT と現欲求を照合し、以下のパターンを検出した場合のみ質問する。

**矛盾検出時**：

- 「F001 の『〇〇』と今回の『△△』は両立しないです。どちらを優先しますか？」

**廃止機能の復活要求検出時**：

- 「F〇〇〇 は YYYY-MM-DD に『〜』を理由に廃止しました。復活条件を満たしますか？」

**却下案の再提案検出時**：

- 「〇〇案は当初『〜のリスク』で却下しました。状況変化はありますか？」

検出なしなら F2 は無言で通過。

### F3: 履歴更新予告（通知形式、承認のみ求める）

質問ではなく通知：

- 「以下を INTENT.md に追加します: [内容]」
- 「F001 の条件を v2 に更新します: [差分]」
- 「R=2 への変更により REGIME.md を更新します」

人間の反応は「承認 / 修正指摘」のいずれか。修正指摘があれば該当箇所のみ直して再提示。

### 曖昧応答が返った場合（E1対応）

人間が「たぶん」「どうだったかな」「どっちでもいい」等で返した場合、**追加質問をしない**。
AI が過去履歴から最適解を推定し、通知形式で充当する：

- 「〇〇と推定しました。違ったら次回教えてください」

INTENT.md の該当項目に `**確度**: AI推定 (YYYY-MM-DD)` を付与し、次回儀式 F2 で一括確認する。
曖昧応答が2回連続したら儀式レベルを自動 -1 格下げ。

### 儀式拒否への対応（E2対応）

「儀式いらない、すぐ作って」「軽くやりたい」等の拒否発話を検出したら、儀式を**完全スキップ**し REGIME-LOG.md に記録する。
質問を重ねず、L1 に引き継ぐ（L1 が背景で履歴照合を回す）。
拒否が5回連続したら SUMMARY.md に警告常設（人間裁量は最後まで尊重、強制介入なし）。

## NFR（非機能要件）質問例

技術用語（SLA / SLO / PII 等）は避け、日常語で聞く。詳細は `nfr-scoring.md` を参照。

### 性能（P）
- 「ボタン押してから反応が何秒以内じゃないと困る感じですか？」
- 「同時に使う人は何人くらい想定？ピーク時と普段の差は？」

### 可用性（A）
- 「壊れて使えない時間があると、どれくらい困りますか？ 1日？ 1時間？ 1分もダメ？」
- 「夜中や休日も動いている必要がありますか？」

### セキュリティ（S）
- 「どんな人のデータを扱いますか？ 社員？ お客様？ 個人名・住所・お金は含む？」
- 「このシステムが外部から攻撃されたら、何が漏れたり壊れたりしますか？」

### コンプライアンス（C）
- 「法律や業界ルールで『こうじゃないとダメ』というのはありますか？」
- 「監査やチェックを受ける必要がありますか？ 誰から？」

### スケーラビリティ（L）
- 「使う人や扱うデータが 10 倍、100 倍に増えたら同じ仕組みで動きますか？」
- 「海外からも使いますか？ 国内だけ？」

## autonomous_scope 質問例（v5.6.0 追加、dev_mode = autonomous 確定後）

dev_mode が `autonomous` に決定した場合、autonomous-drive 機構（PR 作成 / 多層検証 / 自動 merge / 次 Issue 着手）の運用粒度を選択する 1 問を追加する。**デフォルト = full**（人間 = P1〜P4 のみ）。

```
Q: 自律駆動の度合いを選択してください
   (1) フルオート [デフォルト推奨]
       人間 = P1〜P4 (発案 / ブレスト / 事後確認・評価 / 暴走時介入) のみ
       AI = Issue 精査〜PR 作成〜多層検証〜自動 merge〜次 Issue 着手 まで完全自走
       autonomous_scope: full
   (2) 中度
       自動 merge は無効、PR review/approve は人間が実施（P3 を merge 前に倒す）
       autonomous_scope: merge_gated
   (3) カスタム
       個別に設定（dev-env-spec.md Level C 詳細表で指定）
       autonomous_scope: custom
```

質問しないケース：
- M1 単体モード → `local_only` or `github_assisted` のみ、autonomous_scope は適用外
- LC = 0（新規プロジェクト立ち上げ初回）+ 規模 S が小 → `github_assisted` 推奨で本軸スキップ

質問する非エンジニア向け補助:
- 「人間の手介入は『発案するとき』『ブレストするとき』『事後確認するとき』『AI が暴走したとき止めるとき』の 4 場面だけになります。それで OK ですか？」（fullの説明）
- 「PR の最終 merge ボタンだけは自分で押したい場合は (2) を選んでください」（merge_gated の説明）

## current_focus 質問例（v5.7.0 追加、autonomous-drive 入口側 Issue pickup と連動）

dev_mode `autonomous` 確定時、または振り返り儀式で集中対象の変更を検出した時に確認：

```
Q: 今このプロジェクトで何に集中していますか？
   (1) bug-fix     ：master や直近リリースのバグ修正中
   (2) feature     ：新機能開発中（v5.x.0 minor 等）
   (3) refactor    ：コード整理・命名統一・リファクタ中
   (4) docs        ：ドキュメント改修中
   (5) chore       ：依存更新・CI 整備等の雑務中
```

更新は spec-architect が AI 側で REGIME.md の `## current_focus` セクションに記録（β 半自動）。
人間は発話で方向性を伝えるだけ（philosophy 第 4 条 + 第 7 条 P1/P2 と整合）。

更新タイミング：
- 新規機能着手時（例: 「v5.8.0 の新機能を始めたい」→ type: feature 更新）
- フェーズ切替時（例: 「バグ修正に集中したい」→ type: bug-fix 更新）
- 振り返り儀式時（過去 1 週間の commit 傾向と乖離検出時）

current_focus が設定されていると、autonomous-drive 入口側 Issue pickup で「focus と一致しない Issue は skip」される（一時延期、Issue は close せず）。

## 事故起点プローブ

ユーザーが既存システムの機能変更・追加・確認を要求した際、技術的事実回答より**事故事実の聞き取り**を優先するための儀式プロトコル。

### 発動条件

以下の条件を**両方満たす**場合に必須発動：

1. **LC ≥ 1**（既存プロジェクト）
2. **既存機能への言及**を含む初手発話

#### 既存機能言及の検出キーワード

- 保存・確認・追加・戻す・消える・おかしい
- 「既存」「現状」「今の」「このアプリ」「このシステム」
- プロジェクト固有の機能名・画面名

#### 適用除外

- LC=0（新規プロジェクト）での発話
- 純粋な新規機能構想（既存言及なし）

### プローブ実行手順

#### ステップ 0.5: 必須発問（1問のみ）

発動条件に該当したら、技術回答の前に以下のいずれかを**必ず**発問：

> 「何かきっかけになる出来事がありました？」
> 
> 「最近うまくいかないことありました？」

#### ステップ 1以降: 応答に応じた分岐

**Yes系応答**（「消えた」「壊れた」「困った」等）:
- 事故事実の詳細聞き取りを**優先**
- 技術的事実確認は事故把握後に実施

**No系応答**（「特にない」「試したかっただけ」等）:
- 通常の対話フローに移行
- ritual-protocol.md F2 以降を実行

### ritual-protocol F2 との実行順序

```
事故起点プローブ（ステップ 0.5）
↓
F2: 認識ズレ検出（ステップ 1〜）
↓
通常の対話フロー
```

事故起点プローブは「事故の存在確認」= F2 の前提を埋める役割。F2 は過去 INTENT が存在する場合の認識ズレ検出のため、INTENT 空の場合でも事故は存在し得る。

### 哲学的位置づけ（Council 少数意見の温存）

本プローブは**沈黙する前提の言語化**パターンに分類される：

- **A1（事故起点プローブ）**: 対話冒頭で**ユーザー側の沈黙する前提**（事故の存在）を言語化
- **A3（データ整合性事故履歴）**: 履歴層から**プロジェクト側の沈黙する前提**（事故の存在）を言語化

両者は**同型（沈黙前提の言語化）**だが**対象が異なる**（ユーザー記憶 vs プロジェクト履歴）。この同型性は philosophy.md 第8条候補（v6.0.0）として「事故履歴という外部記憶への harness 依存をどう明文化するか」の議論への伏線となる。

## 多言語プロジェクト判定（Wave 1 候補 6、PR #76）

**起点**: `templates/rules/` 階層化規約導入（DH 流遅延戦略）
**Council 採決**: `council-2026-05-11T05:00:00Z-w1qb03`（recommended A: 言語先取りなし）

L0 対話で多言語プロジェクトを判定し、`templates/rules/<lang>/` の生成可否を確定する質問群。

### 使う技術の数の把握

避ける：「言語スタックは？」「実装言語は何ですか？」

推奨：

- 「使う技術が 1 つだけ？ それとも複数の技術を組み合わせる？」
- 「フロントエンドとバックエンドで違う言語使う？」
- 「サーバー側と画面側でプログラム分かれてる感じ？」

### 言語別ルール採用の意思確認

避ける：「言語別 coding-standards を整備しますか？」「lint config 言語別に分ける？」

推奨：

- 「コーディングの決まりを言語別に分けたい？ それとも共通でいい？」
- 「『Python ではこう書く』『TypeScript ではこう書く』を分けて書きたい？」
- 「最初は共通だけで、後から言語別を足してもいい？」（**DH 流遅延戦略の優しい確認**）

### 判定後の処置

| 回答パターン | `templates/rules/` 生成方針 |
|---|---|
| 単一言語、共通でいい | `common/` のみ充填、`<lang>/` 生成なし |
| 単一言語、その言語別を書きたい | `common/` + `<lang>/` の 2 ディレクトリ生成 |
| 複数言語、共通でいい | `common/` のみ充填 |
| 複数言語、言語別を分けたい | `common/` + 言語数分の `<lang>/` 生成 |
| 「分からない」「後から考える」 | `common/` のみ充填（最も保守的、後から `<lang>/` 追加可能） |

# L0 サブフェーズ選定プロトコル

ステップ 3.5 で実行される動的サブフェーズ選定のロジック定義。
基本 5 問で該当プロジェクトに必要なサブフェーズ（L0-2〜L0-6）を起動判定し、モード（完全 / 簡易 / スキップ）を決定する。

---

## 原則

- 既存 REGIME 判定（S/U/R/NFR 軸）とは**直交軸**として動作する。M1 でも認可は必要、L2 でもデータなしはあり得る
- 全サブフェーズ常時起動ではなく、対話で必要なもののみ起動する（動的選定原則）
- 判定結果は `spec/subphase-manifest.md` に記録する（Phase 1 時点は pre-official 規約、Phase 2 で REGIME.md に統合予定）
- 判定は人間 1 問 1 問の回答で確定せず、**全 5 問の応答を総合**してから提示する（対話の自然さを優先）
- 迷ったら**簡易モード**または**スキップ**を推奨（事後追加プロトコルでの昇格が可能）

---

## 判定フロー

```
[ステップ 3 完了]
    ↓
[基本 5 問を対話で実施（dialog-questions.md の新セクション参照）]
    ↓
[各サブフェーズの起動判定 + モード選定]
    ↓
[判定結果を提示フォーマットで人間に提示]
    ↓
[承認] ── 再判定依頼 → 基本 5 問へ戻る
    ↓
[spec/subphase-manifest.md 生成]
    ↓
[承認されたサブフェーズを依存順に実行]
    L0-2 → (L0-3 ‖ L0-4) → L0-5 → L0-6
    ↓
[全サブフェーズ完了後、ステップ 4 へ]
```

**全サブフェーズがスキップの場合**: `spec/` ディレクトリ自体を生成せず、`subphase-manifest.md` も作成しない。ステップ 3.5 は実質的に何も行わずステップ 4 へ移行する。

---

## 基本 5 問

AI が L0 冒頭の中盤（業務意図把握後、ステップ 4 モード判定前）で投げる質問。非エンジニア語彙を厳守する。詳細な対話例は `dialog-questions.md` の「L0 サブフェーズ選定質問」を参照。

| # | 質問 | 判定対象 | 典型分岐 |
|---|---|---|---|
| S1 | 「データを保存する必要がありますか？ DB 使いますか？」 | L0-2 | DB あり=完全 / メモリ複雑=簡易 / ステートレス=スキップ |
| S2 | 「外部のシステムや API とつなぐ必要がありますか？」 | L0-3 | 外部公開=完全 / 内部のみ=簡易 / CLI=スキップ |
| S3 | 「画面はいくつありますか？ 遷移は複雑ですか？」 | L0-4 | 3 状態以上=完全 / 単純=簡易 / 状態なし=スキップ |
| S4 | 「複数ユーザーで使いますか？ 権限の違いはありますか？」 | L0-5 | 所有権あり=完全 / ロールのみ=簡易 / 単独=スキップ |
| S5 | 「時間経過や承認で状態が自動的に変わるものはありますか？」 | L0-6 | 2 サブフェーズ以上起動で自動 ON |

**詰め込み禁止**: 5 問は一度に投げず、S1+S2 / S3+S4 / S5 など 2〜3 問に分けて投げる。既存 `dialog-questions.md` の「1 回のターンで質問を詰め込みすぎない」原則を継承。

## サブフェーズ起動判定表

### L0-2 ドメインモデル

| 条件 | 判定 |
|---|---|
| DB 使用あり | **完全モード**（Zod + TypeScript） |
| データ永続化なし（ステートレス） | **スキップ** |
| メモリ上のみだが複雑なデータ構造あり | **簡易モード**（フラットな型定義のみ、値オブジェクト・集約省略） |

### L0-3 API 契約

| 条件 | 判定 |
|---|---|
| 外部公開 API | **完全モード**（TypeSpec） |
| 内部 API（同一アプリ内） | **簡易モード**（関数シグネチャのみ、TypeSpec 不要） |
| CLI ツール・ライブラリ | **スキップ** |

**前提**: L0-2 がスキップの場合、L0-3 は完全モード不可。簡易モード（またはスキップ）のみ。

### L0-4 状態遷移

| 条件 | 判定 |
|---|---|
| 画面 3 状態以上 OR エンティティにライフサイクル（複数状態）あり | **完全モード**（XState + Mermaid） |
| 単一画面のみ AND 単純 CRUD | **簡易モード**（Mermaid のみ、XState 不要） |
| 状態概念なし | **スキップ** |

### L0-5 認可

| 条件 | 判定 |
|---|---|
| マルチユーザー AND リソース所有権あり | **完全モード**（OpenFGA） |
| ロールベースのみ（ユーザー種別のみ） | **簡易モード**（マトリクス表のみ、OpenFGA 不要） |
| シングルユーザー | **スキップ** |

### L0-6 層間不変条件

| 条件 | 判定 |
|---|---|
| 2 サブフェーズ以上起動（L0-2〜L0-5 のうち） | **起動**（Gherkin Happy/Sad/Evil） |
| 1 サブフェーズのみ起動 | **スキップ**（層間整合の概念が成立しない） |

**特殊性**: L0-6 は他サブフェーズの起動数で自動判定される。S5 の回答は L0-6 の**起動するか**の最終決定ではなく、**どのような不変条件を引き出すか**の対話準備に用いる。

## モード選定サマリ表

| サブフェーズ | 完全モード | 簡易モード | スキップ |
|---|---|---|---|
| L0-2 | DB + 複雑データ | メモリ上の複雑データ | ステートレス |
| L0-3 | 外部公開 API | 内部 API | CLI / ライブラリ |
| L0-4 | 3 状態以上 / ライフサイクル | 単純遷移 | 状態なし |
| L0-5 | リソース所有権あり | ロールのみ | シングルユーザー |
| L0-6 | 2 以上起動時に自動 ON | — | 単独起動時 |

## 判定結果の提示フォーマット

AI が人間に提示する判定結果のテンプレート。

```
=== L0 サブフェーズ選定結果 ===

L0-1 業務意図      : 起動済み（SPEC.md 生成済み）
L0-2 ドメインモデル : 起動・完全モード
  理由: S1 で「PostgreSQL 使用」「複雑なデータ構造あり」と回答
L0-3 API 契約       : 起動・簡易モード
  理由: S2 で「内部 API のみ、外部公開なし」と回答
L0-4 状態遷移      : 起動・完全モード
  理由: S3 で「4 画面構成、TODO のライフサイクル」と回答
L0-5 認可          : 起動・完全モード
  理由: S4 で「マルチユーザー、所有権ベース」と回答
L0-6 不変条件      : 起動
  理由: L0-2/L0-3/L0-4/L0-5 の 4 サブフェーズが起動、層間整合性チェック必要

実行順: L0-2 → (L0-3 ‖ L0-4) → L0-5 → L0-6

この判定でよろしいですか？（修正があれば指摘してください）
```

**曖昧回答時**: `ritual-protocol.md` の E1 対応に従い、「AI 推定 + 通知型」で埋める。例:
> 「DB 使用ありと推定しました（S1 の『たぶん』から推定）。違ったら指摘してください」

推定による判定箇所は manifest で `確度: AI推定` が付与される。

## 事後追加プロトコル

プロジェクト進行中に判定を変更する場合の手順。

### ケース 1: スキップしたサブフェーズを途中で追加

例: 最初は社内ツールで L0-5 スキップ → 外部公開することになり L0-5 必要。

**手順**:
1. 人間が追加を宣言（「認可が必要になった」）
2. AI が依存関係チェック（L0-5 追加なら L0-2/L0-3 の成果物を参照できることを確認）
3. AI が L0-5 対話（Phase α〜δ）を実行
4. AI が L0-6（起動済みなら）を再生成（既存条件 + 認可関連の Evil Path を追加）
5. `subphase-manifest.md` の「事後変更ログ」に記録

### ケース 2: モード昇格（簡易 → 完全）

例: L0-4 簡易モードで始めたが画面が増えて XState 必要。

**手順**:
1. AI が簡易モード成果物（Mermaid のみ）を読み込む
2. 完全モード対話（XState 固有の質問: 並行状態・after トリガー等）を実行
3. 既存 Mermaid と整合する XState JSON を生成
4. 差分を人間にレビュー
5. 既存成果物（`state-machine.ts` / `state-diagrams.md`）を上書き
6. L0-6 を再生成（新規イベント名が Scenario に反映される）

### ケース 3: 誤判定の訂正

例: L0-5 をスキップと判定したが後で必要と判明。

**手順**: ケース 1 と同じ。加えて:
- `subphase-manifest.md` の確度欄を `誤判定訂正` に更新
- Phase 5 導入後は REGIME-LOG.md に「判定ミス」として記録し、次プロジェクトの判定精度向上に活用

## 既存 REGIME 判定との独立性

本選定は `regime-assessment.md` の S/U/R/NFR 判定と**直交軸**である。根拠:

| REGIME 判定 | サブフェーズ起動 | 例 |
|---|---|---|
| M1（S+U+R ≤ 3） | 全起動あり得る | 小規模だが DB+API+認可のある個人開発 |
| M2（R ≥ 2） | 全スキップあり得る | リスク高だが純粋な CLI ツール |
| L2（>15k tok） | 任意 | 大規模でも DB なしの並列バッチ処理 |

REGIME がプロジェクトの**規模・不確実性・リスク**を測るのに対し、本選定はプロジェクトの**構造的特性**（データ層・API 層・遷移・認可の有無）を測る。両者は独立に判定・記録される。

Phase 2 で `regime-assessment.md` 改訂時、REGIME.md に本 manifest が統合される（独立ファイル → REGIME.md の新セクション「L0 サブフェーズ選定」へ）。

## 非該当プロジェクトでの省略

以下のプロジェクトでは基本 5 問自体を省略し、ステップ 3.5 を完全にスキップできる:

- 使い捨て CLI ツール（SPEC.md に「CLI ツール」「ワンライナー」等と明示）
- 実験プロトタイプ（REGIME の U ≥ 3 かつ SPEC.md が機能 1 個未満）
- ドキュメント生成ツールのみ

判断基準: S1〜S5 の全てが「いいえ」または「該当なし」になることが事前に明らかな場合。迷ったら基本 5 問を実施する（時間コストは 2〜3 分）。

## 誤判定のフィードバックループ

Phase 5（`history-layer-spec.md` 改訂）で導入予定。Phase 1 では以下の形式で `subphase-manifest.md` に人間が随時記録する。

```markdown
## 事後評価（プロジェクト完了時）

- L0-2: 判定=完全 / 実績=妥当
- L0-3: 判定=簡易 / 実績=完全必要だった（途中で昇格）
  - 学び: 「外部 API 予定なし」でも将来の開放を考慮して完全モード推奨
- L0-4: 判定=スキップ / 実績=妥当
```

これを Phase 5 で REGIME-LOG.md に集約し、判定表（本ファイル）の再設計材料とする。

## プロトコル自己評価

本プロトコルは以下のシグナルで再設計対象となる:

- 誤判定率が 30% を超える → 判定表の条件見直し
- 事後追加頻度が高い（プロジェクトの 50% 超） → 基本 5 問の質問文見直し
- L0-6 起動率が異常に低い（単一サブフェーズ起動が多発） → 基本 5 問の S5 の再設計、もしくは L0-6 起動条件の緩和
- 人間が判定結果を毎回大幅修正する → 提示フォーマットの見直し、もしくは AI 推定ロジックの改善

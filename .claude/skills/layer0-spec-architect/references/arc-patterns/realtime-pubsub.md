# ARCパターン: realtime-pubsub（リアルタイム pub/sub）

リアルタイム配信・大量同時接続・双方向通信が要件のプロジェクト向けパターン。
社内版 LINE 型、チャット、通知配信、ダッシュボードのライブ更新、IoT テレメトリ等で採用。

---

## 概要

イベント発行者（publisher）とイベント受信者（subscriber）を pub/sub ブローカーで疎結合化し、
WebSocket / SSE / MQTT 等で接続中クライアントへ push 配信する構成。
状態保存（履歴・既読管理・永続化）は別途 DB 側で持つ。

## 適用条件

以下のいずれか1つ以上が該当する場合:
- 同時接続数 ≥ 数千（バースト含む）
- メッセージ配信レイテンシ p95 < 500ms が要件
- 双方向通信（クライアント → サーバ → 他クライアント）
- オフラインクライアントへの未配信分再送が必要
- デバイス／ブラウザ push 通知と統合する

## 必要な技術スタック例

| レイヤー | 代表技術 |
|---|---|
| 接続層 | WebSocket（Socket.IO / ws）/ SSE / MQTT |
| ブローカー | Redis Pub/Sub / NATS / Kafka / RabbitMQ / MQTT broker |
| 配信ゲートウェイ | Centrifugo / Ably / Pusher / Firebase Realtime / Supabase Realtime |
| 状態 DB | PostgreSQL + Redis（オンライン状態キャッシュ） |
| push 通知 | APNs / FCM |

## NFR スコアとのマッピング

| 軸 | 想定スコア | 理由 |
|---|---|---|
| P 性能 | 2-3 | 配信レイテンシ要件が定量化される |
| A 可用性 | 2-3 | 接続が切れる=機能停止扱いのため高め |
| S セキュリティ | 1-2 | 認証必須、メッセージ所有者チェック等 |
| C コンプライアンス | 0-2 | 扱うデータ次第 |
| L スケーラビリティ | 2-3 | 水平スケール前提、接続数に応じたノード追加 |

**典型 N = 7-11** 帯域。N ≥ 9 でモノリス強制解除 → 本パターン採用の主要ルート。

## 実装時の制約とトレードオフ

### 制約
- ブローカーと接続ゲートウェイは**水平スケール前提**で設計する。Sticky session / コンシステントハッシングを明記
- 配信保証（at-least-once / at-most-once / exactly-once）を仕様段階で決定する
- 未読管理・オフライン再送・既読同期は**状態 DB 側の設計問題**であり、pub/sub 層では解決しない
- 接続切断時のリトライ戦略（指数バックオフ、最大リトライ数）を規定する

### トレードオフ
- **メリット**: 低レイテンシ、スケール性、ブローカー差し替え容易性
- **デメリット**: 運用複雑度高、単体テストが難しい、障害点が増える（接続層・ブローカー・DB）、メッセージ順序保証が限定的

## L1 / L2 での扱い方

### L1 autonomous-dev
- 単一ドメイン（チャット機能のみ等）なら M2 で完遂可
- ただし推論的センサーに**配信レイテンシ**と**再接続時の重複排除**の検証観点を追加必須

### L2 orchestrator
- L2 判定の典型例（チャット × 通知 × 既読 × 検索等で独立ドメイン ≥ 5）
- 配下の各サブドメインは monolith でも可。ブローカー層は**共通基盤**として DOMAINS.md に明記

### 追加センサー項目

`sensors/computational.md` に以下を追記する前提:

- 接続ゲートウェイのロードテスト（想定同時接続数の 120% で疎通確認）
- ブローカー障害時のフェイルオーバー検証（手動トリガー）

`sensors/inferential.md` に以下を追記:

- 配信レイテンシ p95 が SPEC 記載の SLO を満たすか
- 切断 → 再接続時に同一メッセージが二重配信されないか（at-most-once 要件のとき）
- 接続中ユーザ数とブローカー側 subscriber 数の整合

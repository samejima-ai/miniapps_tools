# ARCパターン: event-sourcing（イベントソーシング）

状態を直接 UPDATE せず、発生した事象（イベント）の append-only なストリームを唯一の真実として保持するパターン。
金融取引、医療記録、監査必須業務、後からの状態復元・検証が要件のプロジェクトで採用。

---

## 概要

ドメインの全変更は不変なイベント（"OrderPlaced", "PaymentCompleted" 等）として event store に追記され、
現在状態は**イベントストリームを畳み込む（fold）ことで派生**する。
読み取り用のプロジェクション（CQRS の Read Model）と書き込み用のイベントストアを分離する構成を取ることが多い。

## 適用条件

以下のいずれかに該当する場合:
- 監査要件が厳格（金融 / 医療 / 公共 / 規制業界）
- 過去の任意時点の状態を復元できることが要件
- 複雑なビジネスルール変更に対し、再計算で遡及対応したい
- 複数サブシステム間で「何が起きたか」を共通の事実として共有する必要がある
- Temporal / 時系列分析が業務の中核

## 必要な技術スタック例

| レイヤー | 代表技術 |
|---|---|
| event store | EventStoreDB / Kafka（compacted topic）/ PostgreSQL（events テーブル）/ DynamoDB Streams |
| プロジェクション | PostgreSQL / Elasticsearch / 専用読み取り DB |
| イベントバス | Kafka / NATS JetStream / AWS Kinesis |
| スキーマレジストリ | Confluent Schema Registry / Apicurio |
| フレームワーク | Axon / EventStoreDB クライアント / eventuous / 自作 |

## NFR スコアとのマッピング

| 軸 | 想定スコア | 理由 |
|---|---|---|
| P 性能 | 1-2 | append 性能は高いが、プロジェクション再構築時は重い |
| A 可用性 | 2-3 | 金融・医療では 99.95% 以上が前提 |
| S セキュリティ | 2-3 | PII・決済・医療情報を扱うケースが主 |
| C コンプライアンス | **2-3** | 本パターン採用の主要動機。監査要件が前提 |
| L スケーラビリティ | 1-3 | 読み書き分離で高スケール、単一 aggregate 書込は直列 |

**典型 N = 9-14** 帯域。N ≥ 9 でモノリス強制解除 → 本パターンが有力候補になる。

## 実装時の制約とトレードオフ

### 制約
- **イベントスキーマは不変**。スキーマ変更は upcasting / versioning で対応（`schema-evolution.md` を参照）
- 既存イベントを**削除・書き換えしない**。訂正イベント（CompensatingEvent）で対処
- プロジェクション再構築に必要な時間を想定し、**スナップショット戦略**を規定する（N イベントごと／時刻ごと）
- 最終的整合性（eventual consistency）前提。UI は Read Model と書き込み結果の乖離を考慮した設計に

### トレードオフ
- **メリット**: 監査完備、時系列復元、業務ルール遡及変更、デバッグ性（何が起きたかが全て残る）
- **デメリット**: 学習コスト高、初期実装重い、CRUD 的思考から遠い、プロジェクション再構築に時間、AI の自走が難しい領域多数

## L1 / L2 での扱い方

### L1 autonomous-dev
- **単純なドメインでも M2 必須**。独立検証で aggregate 境界・イベント定義の妥当性チェックが必要
- 自走完遂できる範囲は「1 aggregate + 数プロジェクション」程度に限定
- それ以上は L2 に委譲する判断を Layer 0 で先に済ませる

### L2 orchestrator
- event-sourcing の典型運用形。aggregate 単位でサブドメイン分割
- **共通の event schema registry** と **共通の event bus** は layer2-integration-verifier で跨ぎ検証必須
- DOMAINS.md にイベント名一覧・スキーマバージョン・発行/購読関係を明記

### 追加センサー項目

`sensors/computational.md` に以下を追記する前提:

- イベントスキーマの互換性チェック（`schema-evolution.md` のポリシーに従う）
- プロジェクション再構築のドライラン（イベントストリーム再生 → 想定 Read Model 生成の検証）

`sensors/inferential.md` に以下を追記:

- aggregate 境界がトランザクション整合性境界と一致しているか
- 訂正イベントで過去事実が「上書き」ではなく「追記」で表現されているか
- プロジェクションが最終整合性前提で設計されているか（UI 側の扱いも含む）
- 監査要件が満たされているか（誰が・いつ・なぜを全イベントが保持）

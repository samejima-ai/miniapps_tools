# ARCパターン: monolith（モノリス）

デフォルトのアーキテクチャパターン。ARC 未指定時はこれが選択される。
単一デプロイ単位・単一プロセス・単一データストアを原則とする。

---

## 概要

1つのアプリケーションコードベースを1つのデプロイメント単位として運用する。
機能追加はモジュール追加として行い、プロセス境界を越えた通信は基本的に発生しない。

## 適用条件

- 機能数 ≤ 20 程度
- 独立ドメイン数 ≤ 5
- N ≤ 4（NFR 軸合計）
- 単一チーム、単一デプロイライン
- トランザクション整合性を DB 1インスタンスで担保できる
- AI が自走開発で最後まで到達できる最もシンプルな形

## 必要な技術スタック例

| レイヤー | 代表技術 |
|---|---|
| Web | Next.js / Rails / Django / Laravel / Spring Boot |
| DB | PostgreSQL / MySQL（単一インスタンス＋リードレプリカ程度） |
| キャッシュ | Redis（任意、セッション用） |
| ジョブ | DB ベースキュー / cron ベーススケジューラ |
| 配信 | 単一サーバ or 単一コンテナ |

## NFR スコアとのマッピング

| 軸 | スコア | 適性 |
|---|---|---|
| P 性能 | 0-2 | ○（単一 DB の縦スケール内で足りる範囲） |
| A 可用性 | 0-2 | ○（99.9% 相当まで）、3 は非推奨 |
| S セキュリティ | 0-2 | ○（PII まで対応可） |
| C コンプライアンス | 0-2 | ○（業法準拠まで） |
| L スケーラビリティ | 0-1 | ○（水平スケール前提なら他パターンへ） |

**合計 N ≤ 4** で採用可能。N ≥ 5 で推奨切替、N ≥ 9 で強制解除。

## 実装時の制約とトレードオフ

### 制約
- バックグラウンドジョブと Web を同一プロセスに同居させる場合、長時間処理がリクエストを阻害しないよう明示分離（例: Sidekiq/RQ を別プロセスで起動）
- スキーマ変更は無停止マイグレーション戦略（後方互換のある ADD 系から）を前提にする
- ログ・メトリクスは外部サービスへ逃がす（本体の可用性に巻き込まれないため）

### トレードオフ
- **メリット**: 開発速度、運用簡素、トランザクション一貫性、AI 単独完遂可能性
- **デメリット**: 部分スケール不能、言語選択の固定化、デプロイ単位が大きい

## L1 / L2 での扱い方

### L1 autonomous-dev
- 通常パス。追加センサー不要
- 1 デプロイ単位として動作検証（build + 型 + テストで完結）

### L2 orchestrator
- 通常 L2 判定では選ばれない（L2 は独立ドメイン ≥ 5）
- ただし**サブドメインの内部実装として monolith を採る**ことは頻出（L2配下の各 L1 サブは monolith デフォルト）

### 追加センサー項目
monolith 固有の追加センサーは**なし**（`sensors/computational.md` と `sensors/inferential.md` のデフォルトのみで充足）。

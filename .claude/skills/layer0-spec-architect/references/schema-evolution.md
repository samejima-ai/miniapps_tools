# スキーマ進化プロトコル

データモデル（DB スキーマ、API、メッセージスキーマ、永続イベント）の変更を**破壊的変更なしに**進化させるための規格。
非該当プロジェクト（プロトタイプ、使い捨てツール等）では SPEC.md の「データモデル進化」セクションを省略可。

---

## 適用対象

以下のいずれかに該当する場合、本プロトコルの遵守を強く推奨:

- 本番運用中で既存レコードが存在する
- 外部クライアント（モバイル／SDK／API 利用者）がスキーマに依存する
- ARC が `event-sourcing` に確定している（**完全準拠必須**）
- ARC が `realtime-pubsub` でメッセージスキーマを公開している（推奨）
- ARC が `monolith` でも、本番 DB マイグレーションで無停止デプロイが要件

---

## 後方互換性ポリシー（3 種）

### A. full-compat（完全互換・デフォルト）

新バージョンが旧バージョンの読み書きを完全サポートする。

- 追加は任意（必須でない）カラム／フィールドのみ
- 削除は禁止（非推奨マークのみ）
- 既存フィールドの型変更・セマンティクス変更は禁止
- デフォルト値は旧バージョン読み取り時に適用される

### B. read-compat（読み取り互換）

新バージョンが旧データを読める。旧バージョンは新データを読めない。

- 必須フィールドの追加可（ただし旧データには upcast でデフォルト注入）
- 読み取り時マイグレーション（lazy migration）許容
- ライター側を先にデプロイし、一定期間後にスキーマをクローズ

### C. breaking（破壊的）

双方向互換を放棄する。前提として全クライアントの一斉更新が必要。

- バージョン番号をメジャー更新（v1 → v2）
- 旧バージョンへのリクエストは明示的エラー or 別エンドポイント
- event-sourcing では**採用不可**（既存イベント再生が壊れる）

---

## デプロイ戦略（3 種）

### 1. expand-contract（拡張後収縮）

1. **expand**: 新スキーマを追加。旧スキーマは残す。両方書き込む（dual-write）
2. **backfill**: 既存データを新スキーマに移行
3. **switch-read**: 読み取りを新スキーマに切り替え
4. **contract**: 旧スキーマを削除（十分な待機期間後）

- リスク低。無停止可能
- デフォルトでこの戦略を採る

### 2. blue-green schema

- DB 2 系統を並行運用し、アプリ側でルーティング切替
- 大規模変更時。運用コスト高
- monolith では稀。realtime-pubsub / event-sourcing で pub/sub 層の切替に有効

### 3. versioned-endpoint

- API / イベントにバージョンを埋め込み、複数バージョンを同時提供
- 外部公開 API で旧クライアント切り捨てが不可能なケース
- モバイルアプリとの整合維持で頻出

## event-sourcing での完全準拠

ARC = event-sourcing の場合、以下を**必須**とする（`arc-patterns/event-sourcing.md` の制約とも整合）:

### 必須ルール
- **既存イベントの書き換え・削除は絶対禁止**。訂正は CompensatingEvent を追記
- スキーマ変更は **upcasting**（読み取り時に旧バージョンを新バージョンへ変換）で対応
- イベント型にバージョン番号を必須付与（例: `OrderPlaced.v2`）
- スキーマレジストリ（Confluent Schema Registry / Apicurio 相当）でスキーマを集中管理
- プロジェクション再構築時に全バージョンが解釈可能であること

### 禁止事項
- 既存イベントへの必須フィールド追加（upcast で初期値を注入できる場合のみ許容）
- イベント型の rename（deprecate して新型へ移行）
- event store への DELETE / UPDATE クエリの発行

## 正規化・非正規化と分析スキーマ（v5.17.0 追加）

OLTP（トランザクション処理）と OLAP（分析処理）でデータ設計の最適点が異なるため、用途に応じて意図的にルールを崩す判断軸を明文化する。

### 正規化（1NF〜3NF）— OLTP デフォルト

- **1NF**: 各列は不可分（配列・繰り返しグループ禁止）
- **2NF**: 1NF + 部分関数従属の排除（複合キーの一部にのみ依存する列を分離）
- **3NF**: 2NF + 推移関数従属の排除（非キー列同士の依存を分離）

L0-2 で生成された `spec/domain.ts` が物理スキーマに展開される際、L1 はデフォルトで 3NF を目指す。データ重複を排除し、更新異常（同じ事実の複数箇所更新による不整合）を防ぐ。

### 意図的非正規化

**判断基準**: 読み取り頻度が書き込み頻度を大きく上回り、JOIN コストがボトルネックの場合のみ採用する。

| 判断軸 | 正規化を維持 | 非正規化を採用 |
|---|---|---|
| 主な操作 | 書き込み・更新中心（OLTP） | 読み取り中心（OLAP / レポート / ダッシュボード） |
| データ一貫性 | 強整合性が必須 | 結果整合性で許容（`arc-patterns/event-sourcing.md` 参照） |
| クエリパターン | 多様な JOIN | 固定的な集計クエリ |
| 更新頻度 | 高 | 低 or バッチ更新可 |

非正規化を採用する場合、SPEC.md の「データモデル進化」セクションに **更新責務の所在**（どの操作が冗長コピーを更新するか）を明記する。これを怠ると更新異常が必ず再発する。

### スター・スノーフレークスキーマ — DWH 用途

データウェアハウス（DWH）/ 分析基盤を別途構築する場合、OLTP スキーマとは独立にディメンション化された分析スキーマを用意する：

- **ファクトテーブル**: 事実（売上・クリック・イベント等の数値計測）
- **ディメンションテーブル**: 文脈（時刻・地域・商品カテゴリ・ユーザー属性）
- **スター**: ファクト 1 + ディメンション複数（最も単純、クエリ高速）
- **スノーフレーク**: ディメンションをさらに正規化（ストレージ効率優先、JOIN 増）

NFR スコア L (Scalability) ≥ 2 + 分析要件ありの場合、ARC = monolith でも DWH 分離を別案件として提案する（実装は対象外、設計示唆のみ）。

## 半構造化フィールド（JSONB / EAV）の採用判断（v5.17.0 追加）

オプション属性が膨大なケースや、属性自体がランタイムで定義される（ユーザー定義カスタムフィールド等）ケースで、従来の列追加では破綻する設計問題への対処。

### JSONB（PostgreSQL JSON カラム型）

**採用シグナル**:
- 属性のうち 30% 以上が NULL になる「疎」な構造
- 外部 API からの構造化レスポンスをそのまま保存（変換コスト削減）
- 検索対象は構造の一部のみ、残りは取得時の付随情報

**弱点と回避**:
- インデックスは GIN が必要、通常の B-tree より遅い
- スキーマ検証はアプリ層（Zod 等）で行う、DB は構造を強制しない
- 頻繁に検索する key は **生成列（Generated Column）** で展開してインデックス対象化

### EAV（Entity-Attribute-Value）

**採用シグナル**:
- 属性が**ランタイムでユーザーにより追加**される（カスタムフォーム / メタデータ駆動 UI）
- 属性の型が動的（テキスト・数値・日付の混在をユーザーが選択）

**深刻な弱点**:
- クエリ性能が悪い（属性ごとに JOIN）
- 型安全性をアプリ層で完全自前管理
- スキーマ進化（`schema-evolution.md`）の追跡が複雑化

**推奨代替案**:
- 属性集合が事前に列挙可能なら JSONB
- 真の動的属性が必要でも、よく使われる属性は別列に昇格させるハイブリッド構成

### 判定フロー

```
1. 属性集合は静的か？
   YES → 通常の列定義（正規化）
   NO  → 2 へ
2. ランタイムで属性自体が追加されるか？
   NO  → JSONB（疎な属性、外部 API レスポンス保存）
   YES → 3 へ
3. クエリは属性別の集計が必要か？
   NO  → JSONB + 生成列でカバー可
   YES → EAV（最終手段、アプリ層で型安全性を完全自前管理）
```

JSONB / EAV を採用する場合、ファイル冒頭の「適用対象」基準（本番運用 / 外部クライアント / event-sourcing 等）に該当しなくても、本プロトコルの **追加ガイダンスとして** 以下を `SPEC.md` データモデル進化セクションに明記する（適用対象リスト自体は変更しない）：

- 構造変化のバージョニング戦略
- 旧構造の読み取り互換性（read-compat / upcast）
- 検索インデックス（GIN / 生成列）の追従責務

採用プロジェクトが「適用対象」基準に該当しないなら、上記のみを記述し、後方互換性ポリシー（A/B/C）・デプロイ戦略（expand-contract 等）の全項目記入は不要。

## プロトコル自己評価

体制事後評価で以下を蓄積:

- breaking 変更が発生した場合、事前に非該当プロジェクト判定をしていたか
- expand-contract の contract を忘れて旧スキーマが残り続けていないか
- event-sourcing プロジェクトで upcast 以外の手段が使われていないか
- 非正規化を採用した箇所で更新責務が明記されているか（漏れると更新異常が必ず再発）
- JSONB / EAV 採用時に GIN / 生成列のインデックス追従が L1 実装で達成されているか
